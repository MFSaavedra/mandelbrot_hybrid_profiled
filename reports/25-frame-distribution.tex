\begin{abstract}
This report adds the first multi-machine axis to the renderer: frames of the
interpolated sequence are distributed \emph{across} nodes by a static
cyclic/block-cyclic predicate (three environment variables, one line in the
frame loop), while the region-level work queue stays intra-node and
untouched. Correctness is settled first: on the production spec (100 frames,
$1920\times1080$, \texttt{diffT}\,=\,0.1), every executor-pinned check ---
three simulated local nodes in CPU-only cyclic, GPU-only cyclic, and CPU-only
block-cyclic(8), plus a real two-machine run over SSH --- reproduces the
single-process output \textbf{100/100 frames byte-identical}, including
across compilers and microarchitectures (g++-15/Coffee~Lake vs.\
g++~16/Haswell). The benchmark is the honest half: against a laptop-alone
baseline of $18.75\pm0.34$~s, \emph{every} two-node static configuration
loses --- best $28.24\pm0.58$~s ($+51\%$), worst $65.12$~s --- because the
second node is $4.56\times$ slower per frame and equal shares strand the wall
on it. The ceiling is analytic: a perfectly weighted split would reach only
$15.4$~s ($1.22\times$), and the measured $3.4$--$4.2$~s ship/collect
overhead consumes most of that. Static distribution needs near-homogeneous
nodes; a slow node buys capacity, not latency.
\end{abstract}

\section{Motivation: the partition boundary is where prediction fails}
\label{sec:motivation}

Reports~12--23 exhausted most of the single-machine levers: the wall fell
from $\sim$53~s to 23.7~s through affinity and work reduction, and the CPU
pool binds at $<$1.4\% thread spread in every regime
(\texttt{13-zoom-points.tex}). The next multiplier is more machines --- but
\emph{not} by distributing the region queue. Regions are millisecond-scale
tasks (median $\sim$10~ms post-periodicity), so network dispatch latency
would be a first-order cost, and region cost is unknowable before sampling
--- that unpredictability is the entire reason the intra-node scheduler is
dynamic. Any static partition below the frame level inherits the
misprediction problem the queue exists to solve.

The frame is therefore the unit of distribution: the two-level hierarchy
assigns frames across nodes statically and leaves within-frame scheduling
dynamic. What makes the static assignment non-trivial is the $32\times$
per-frame cost spread measured in \texttt{13-zoom-points.tex}
(2.37--76.3~s), trending upward along the zoom. A contiguous block
assignment strands whichever node draws the deep block; cyclic assignment
(node $i$ renders frames $i, i{+}N, i{+}2N, \ldots$) hands every node a
representative cheap/expensive mix. Block vs.\ cyclic vs.\ block-cyclic is
exactly the static-LB study report~13 earmarked; \S\ref{sec:analysis}
delivers its first measured pass --- on deliberately heterogeneous hardware.

\section{The change: one predicate, three environment variables}
\label{sec:change}

Commit \texttt{aae49fb} adds \texttt{DIST\_NODES} / \texttt{DIST\_RANK} /
\texttt{DIST\_BLOCK} (environment variables; unset $=$ off, byte-for-byte
the old behaviour). The entire distribution decision is one line in
\texttt{main.cpp}'s frame-construction loop:

\begin{sourcecode}{cpp}{main.cpp --- the frame-ownership predicate (commit \texttt{aae49fb}, abridged).}
bool owned = (distNodes <= 0) || ((i / distBlock) % distNodes == distRank);
if (owned)
  {
    // fname keeps the GLOBAL frame index i, so the union of all
    // nodes' outputs is collision-free with no renumbering.
    sprintf (fname, "%s%04i.png", imageFilePrefix, i);
    fr[i] = new MandelFrame (uX, uY, lX, lY, resolutionX, resolutionY, fname, iter);
    fr[i]->frameIndex = i;
    workQ.append (new MandelRegion (uX, uY, lX, lY, 0, 0,
                                    resolutionX, resolutionY, fr[i]));
  }
else
  fr[i] = NULL;       // skipped frame: no buffer, no root region
// The interpolation ACCUMULATES (uX += sx1, ...): it must advance
// for every i, owned or not, so frame i's corners stay bit-identical
// to a single-node run (base + i*step would round differently).
uX += sx1;  uY += sy1;  lX += sx2;  lY += sy2;  iter += iterInc;
\end{sourcecode}

Three properties carry all the correctness weight. \textbf{(a)~The
interpolation accumulation is untouched.} Frame $i$'s corners are $i$
sequential FP additions; in IEEE~754, $(a{+}s){+}s \neq a{+}2s$ in the last
ulp, so computing corners as \texttt{base + i*step} --- or running the loop
only over owned indices --- would seed every pixel's orbit differently.
Ownership gates only the allocation and the enqueue. \textbf{(b)~Filenames
carry the global frame index}, so multi-node collection is a plain file
union: ownership is a partition (each $i$ satisfies the predicate for
exactly one rank), hence no collisions and no renumbering; a gap in the
union is immediately visible. \textbf{(c)~Nothing downstream knows the
feature exists.} The workers drain whatever roots were enqueued; each frame
saves itself when its own atomic counter reaches zero; the colormap seed and
\texttt{CUDAmemSetup} depend only on the spec endpoints and resolution. A
side effect: per-node frame-buffer memory drops to owned frames only
($\sim$1/N of the $\sim$830~MB full sequence).

\texttt{DIST\_BLOCK} makes the predicate cover the whole static-assignment
family with one knob: block $=1$ is pure cyclic; block $=k$ is block-cyclic;
block $\geq \lceil \mathrm{frames}/N \rceil$ degenerates to contiguous
block.

\paragraph{The manager-worker seam (deliberately not built).} Static
assignment cannot absorb heterogeneous node speeds or a mispredicted monster
frame; the agreed future fix is a dispenser --- finish a frame, request the
next index, one message per frame. The predicate is the seam: it is the only
place that decides whether this process renders frame $i$.
\texttt{WorkQueue::append} is already mutex-safe for incremental insertion;
the one obligation a dispenser inherits is that \texttt{CalcThr::run} exits
on a momentarily empty queue, so the next root must be enqueued before the
current frame drains. \S\ref{sec:ceiling} quantifies exactly how much a
dispenser would be worth on the pair measured here.

\paragraph{Orchestration.} \texttt{scripts/dist\_frames.sh} drives one job
per rank with GNU~parallel over SSH: ship the (single, shared) spec, run in
a per-rank working directory with the \texttt{DIST\_*} variables set inline,
collect the PNGs back by their global names. Hosts-file lines are
\texttt{host [rdir [args...]]}, so each node carries its own project root
and run configuration --- the heterogeneity hook used below. The run
\texttt{cd}s into the working directory rather than encoding node paths into
the image prefix, side-stepping the known \texttt{char[42]} prefix overflow
(\texttt{21-periodicity-check.tex}).

\paragraph{The CPU-only build (commit \texttt{52c40c4}).} Node~B's GT~750M
is Kepler \texttt{sm\_30} --- dropped after CUDA~10.2, and at $\sim$30
GFLOPS FP64 (1:24 ratio) it would lose to its own CPU on this FP64-bound
kernel (\texttt{16-ncu-divergence.tex}). \texttt{make GPU=0} therefore
skips \texttt{nvcc} entirely: \texttt{kernel\_stub.cpp} satisfies
\texttt{kernel.h}'s three \texttt{extern "C"} symbols and fails loudly if
\texttt{gpuEnable=1} is forced; \texttt{-DNO\_NVTX} replaces the NVTX header
with no-op inlines; the system g++ compiles and links (the g++-15 pin exists
only for \texttt{nvcc} compatibility).

\paragraph{Identity checks (\texttt{verify\_dist.sh}).} Byte-identity is
only well-defined with the executor pinned: the CPU and CUDA
\texttt{diverge()} differ in last-ulp FP64 rounding on deep-frame boundary
pixels ($\sim$8\% of pixels GPU-vs-CPU, \texttt{20-igpu-opencl.tex}), and
the hybrid's region$\to$executor assignment is timing-dependent --- a
pre-existing single-node property, not something distribution introduced.
The checks therefore mirror experiments 21 (CPU-only \texttt{cmp}) and 23
(GPU-only): $N=3$ simulated nodes on the laptop (hosts ``\texttt{:}'',
serialized) in three modes, and one real two-machine run, each compared
frame-by-frame (\texttt{cmp}, exact bytes) against a single-process
reference of the same binary and configuration.

\paragraph{Benchmark (\texttt{bench\_2node.sh}).} Five timed configurations,
3 reps each: the laptop-alone hybrid baseline; four two-node runs ---
cyclic ($\mathrm{block}=1$), block-cyclic ($\mathrm{block}=8$), contiguous
block ($\mathrm{block}=50$; ivy gets the deep frames 50--99), and the
reversed block (hosts order swapped; ivy gets the cheap frames 0--49) ---
plus one ivy-solo capability rep. Two walls are recorded per run:
\texttt{wall\_e2e\_s} times the whole orchestration (ship $+$ ranks $+$ PNG
collection), and each node's own \texttt{[total\_elapsed\_s]} isolates
compute from plumbing.

\section{Analysis}
\label{sec:analysis}

\subsection{Distribution changes which frames exist, and provably nothing else}
\label{sec:identity}

All four identity checks in Table~\ref{tab:identity} are 100/100 exact ---
300 frame comparisons across process counts, assignment shapes, and, in the
last row, physical machines. The reason is structural: the predicate gates
only the \texttt{MandelFrame} allocation and the root-region enqueue, while
the two inputs that determine pixel values --- the accumulated corner
interpolation and the \texttt{diverge()} arithmetic --- run identically
whether or not a frame is owned. The cross-machine row is the strong
version of the claim: node~B computes with a different compiler (g++~16
vs.\ g++-15), a different microarchitecture (Haswell vs.\ Coffee Lake), and
a different build flavour (\texttt{GPU=0} stub, \texttt{-DNO\_NVTX}), and
still reproduces every byte. This works because the build uses plain
\texttt{-O2} with no \texttt{-march}: both compilers emit baseline x86-64
scalar SSE2 FP64 with no FMA contraction, so the orbit recurrence rounds
identically instruction by instruction. The implication cuts both ways:
the identity guarantee is real, and it is \emph{fragile} --- adding
\texttt{-march=native} to the CPU flags would break it through FMA
contraction, the same last-ulp mechanism that already separates CPU from
CUDA output.

\subsection{Equal shares strand the wall on the slow node}
\label{sec:equal-share}

The headline: cyclic --- the assignment that \emph{solves} the zoom's cost
trend --- runs $46.90$~s against $18.75$~s alone, $+150\%$. The mechanism is
the throughput ratio in Table~\ref{tab:nodes}: ivy needs $0.855$~s per
production frame where the laptop hybrid needs $0.188$~s ($4.56\times$), so
any assignment that hands both nodes $\sim$50 frames makes the wall
$\max(10.3,\,42.7) = 42.7$~s of compute plus overhead. The laptop finishes
its share and idles for $\sim$76\% of ivy's rank. Block-cyclic(8) is a wash
with cyclic ($45.00$ vs.\ $46.90$~s, $1.6\sigma$ apart): the interleave
shape only redistributes cost \emph{within} each node's share, and both
shares are the same $\sim$50 frames. Interleave cannot fix share size ---
that is the defining limitation of the uniform predicate on non-uniform
hardware.

\subsection{Block orientation is a built-in 70/30 weighting --- and measures the cost trend}
\label{sec:block}

The two contiguous-block runs bracket the whole benchmark: $65.12$~s with
ivy on the deep half, $28.24$~s reversed --- a $2.3\times$ swing from
orientation alone. The halves are not equal in work: the deep half carries
$68.7\%$ of the laptop's solo cost ($12.52$ vs.\ $5.70$~s) and $71.2\%$ of
ivy's ($61.55$ vs.\ $24.86$~s). That two machines with different ISAs and
executors agree on $\approx$70:30 confirms the split is a property of the
workload --- the zoom's cost trend integrated over each half --- not of any
node. The reversed block is therefore a crude weighted assignment (fast
node takes the 70\% half), and it is the best two-node configuration
measured; but even this ``right'' orientation loses to the baseline by
$+51\%$, because a 70/30 work split is still far from the 82/18 split the
throughput ratio demands.

\subsection{The ceiling is analytic: $1.22\times$, and overhead eats it}
\label{sec:ceiling}

No scheduler --- static, weighted, or the future dispenser --- can beat the
combined throughput bound: $1/18.75 + 1/85.48 = 0.0650\;\mathrm{s^{-1}}$,
an ideal two-node wall of $15.4$~s, a $1.22\times$ speed-up, with ivy
carrying 18~frames. The measured orchestration overhead ($3.4$--$4.2$~s:
spec shipping, SSH session setup, and $\sim$50 deep-frame PNGs collected
back over the LAN) consumes most of the ideal $3.4$~s gain before
scheduling quality even enters. The conclusion is not that distribution
failed --- the mechanism does exactly what it claims --- but that this
\emph{pair} has nothing to distribute for latency: a node $4.56\times$
slower contributes at most 18\% of the work, and the honest use of ivy is
capacity (rendering a second spec, or rendering while the laptop does
something else, where its 85~s/spec stands alone) rather than shortening
the laptop's 18.75~s. A dispenser is \emph{necessary} for heterogeneous
pools --- it would find the 82/18 split automatically --- but on this pair
it would be barely \emph{sufficient}, chasing a sub-20\% ideal through
several seconds of fixed overhead.

\subsection{What remains of the report-13 earmark}
\label{sec:earmark}

The block-vs-cyclic question this experiment was built to answer is
deliberately unanswerable on heterogeneous hardware: when one rank binds at
more than twice the other, the interleave's effect ($46.90$ vs.\ $45.00$~s)
is noise against the share-size effect ($28.24$ vs.\ $65.12$~s). What this
pass \emph{does} establish for the homogeneous study: the harness works
end-to-end on real machines, per-rank compute is cleanly separable from the
$3.4$--$4.2$~s orchestration overhead, and the 70:30 half-split quantifies
exactly the cost trend that cyclic assignment exists to neutralize. On
$N$ comparable nodes the prediction is direct: block strands the deep-block
node at $\sim$$N\times0.7\times$ the mean share (the $65.12$~s row is this
effect at $N=2$), while cyclic and block-cyclic should tie within noise ---
block-cyclic(8) trading a slightly coarser mix for fewer cross-frame
working-set switches.

\subsection{A caveat on absolute walls: the baseline moved}
\label{sec:drift}

Today's laptop baseline is $18.75\pm0.34$~s where
\texttt{23-gpu-periodicity.tex} recorded $23.72$~s steady-state on the same
code lineage --- $21\%$ faster, far outside both $\sigma$'s. Nothing in
this branch touches the render path (the identity checks prove it
byte-exact), so the shift is environmental --- thermal headroom and/or
driver drift between measurement days. Every comparison in this report is
same-day and unaffected; the lesson for cross-report arithmetic is that
absolute walls need a same-day baseline, which is why
Table~\ref{tab:bench} carries its own.

\section{Recommendations}
\label{sec:recommendations}

\begin{enumerate}
  \item \textbf{Ship the mechanism; gate nothing on the benchmark.} The
        \texttt{DIST\_*} predicate is 100/100 byte-identical in all four
        checks, costs nothing when unset, and cuts per-node frame-buffer
        memory to owned frames. The negative benchmark result is a property
        of this hardware pair, not of the code.
  \item \textbf{Do not use ivy to shorten production walls.} Ceiling
        $1.22\times$ ($15.4$~s ideal vs.\ $18.75$~s alone) minus
        $3.4$--$4.2$~s overhead is a wash at best. Use ivy for capacity:
        a second spec in parallel is worth its full $85$~s/spec.
  \item \textbf{If a heterogeneous pool must cut latency, build the
        dispenser at the documented seam} (or add a weighted share to the
        predicate). Expected gain on this pair: $18.75 \to \sim$$15.4$~s
        compute before overhead --- measure the overhead first; it is
        currently half the prize.
  \item \textbf{Run the earmarked block/cyclic/block-cyclic study on
        comparable nodes only} (\S\ref{sec:earmark}); on heterogeneous
        hardware the share-size effect buries the interleave effect by an
        order of magnitude. Keep cyclic as the default meanwhile --- it is
        never worse than block-cyclic here and neutralizes the measured
        70:30 trend by construction.
  \item \textbf{Never add \texttt{-march=native} to the CPU flags} while
        cross-machine byte-identity is part of the methodology
        (\S\ref{sec:identity}); FMA contraction breaks last-ulp equality.
        If per-machine tuning is ever wanted, downgrade the identity check
        to the $\pm$1-iteration equivalence used for CUDA-vs-OpenCL in
        \texttt{20-igpu-opencl.tex}.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}

The two-level hierarchy holds up exactly as designed: a one-line static
ownership predicate distributes frames across machines with output
byte-identical to a single-node run --- 100/100 in every executor-pinned
check, including across compilers, microarchitectures, and build flavours
--- because ownership gates allocation, never arithmetic. The first
measured pass of the static-LB study, run on a deliberately heterogeneous
pair, returns a clean negative: with a $4.56\times$ per-frame throughput
gap, every equal-share configuration loses to the fast node alone (best
$+51\%$, worst $+247\%$), block orientation swings $2.3\times$ by handing
one node the 70\% half of the zoom's cost trend, and the analytic ceiling
of $1.22\times$ --- half of it consumed by $3.4$--$4.2$~s of orchestration
overhead --- says no scheduler can rescue this pair for latency. The
mechanism ships; the slow node earns its keep on capacity; the interleave
comparison awaits homogeneous hardware, where the $65$-second block row
already previews the failure mode cyclic assignment exists to prevent.
