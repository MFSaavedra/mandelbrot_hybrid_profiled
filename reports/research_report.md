# Beating the minibrot wall and unlocking arbitrary deep zooms

The renderer is throughput-bound on minibrot-interior tiles and capped at ~1e-15 zoom by double precision, and both problems have well-established cures that map cleanly onto a CPU+GPU work-queue architecture. The single biggest near-term win is **periodicity / cycle detection inside the iteration loop**, which converts the 13–15-second outlier tiles into milliseconds because every minibrot interior pixel enters a short attracting cycle long before maxIter. The single biggest capability gain is **perturbation theory plus bivariate linear approximation (BLA) with Zhuoran rebasing**, which is what every serious deep-zoom renderer built since 2013 uses to push zoom depth from 1e-15 to effectively unlimited while keeping the per-pixel inner loop in hardware doubles on the GPU. Async CUDA streams remain the obvious throughput-side win on the GPU pipeline. Work-stealing, by contrast, is the wrong answer for this codebase. Below is the technical substance of each technique, the actual algorithms and formulas, the renderers that implement them, and a prioritized roadmap calibrated to the developer's profiling findings.

## Why subdivision has run out of room

The developer's diagnosis is correct and worth restating because it determines which techniques are worth pursuing. Adaptive quadtree subdivision cannot help a region whose corner and 9-point stencil samples all hit maxIter, because the region looks perfectly uniform to any sampling probe. The work is real — every interior pixel has to be iterated to some stopping criterion — and subdivision merely redistributes it. The only meaningful levers from here are **reducing total iteration work** (interior certificates and cycle detection), **reducing per-iteration cost** (SIMD, smarter inner loops, GPU offload of the right regions), and **enabling deeper zooms without inflating per-iteration cost** (perturbation theory). The load-balancing layer is already extracting most of what it can: with 2–4% spread across 11 threads, even a perfect scheduler tops out around 1.02–1.04×. The minibrot tiles are tail-latency artifacts, not balance artifacts.

## Periodicity detection: the direct fix for the minibrot outliers

Every pixel in a minibrot interior has an orbit that converges to an **attracting cycle of length p** (the period of that hyperbolic component), with multiplier |F_c^p'(z*)| < 1. Once the orbit enters the cycle, it revisits the same p points forever in exact arithmetic and within a tight epsilon in floating point. Detecting this lets you exit the loop the moment cyclic behavior is confirmed, instead of grinding to maxIter. This is the technique Fractint identified in the early 1990s as one of its "three most important" speed mechanisms, and it is precisely the **general form of the interior certificate** the developer's roadmap keeps pointing at — it fires on minibrots, the main cardioid, the period-2 bulb, and every other hyperbolic component, with no special case per period.

The classical Fractint implementation keeps one saved z value, compares the current z to it on every iteration, and refreshes the saved value on a doubling schedule. **Brent's cycle detection** is the modern preferred form: save z at iteration indices that are powers of two, then compare every subsequent iteration against that saved value until the next power-of-two checkpoint, when the saved value is reset. It uses one function evaluation per step (Floyd's tortoise-and-hare uses two), O(1) memory, and adds essentially one compare and one branch per iteration. Gnofract4D uses Brent's algorithm; Ultra Fractal exposes periodicity checking as a built-in option that is automatically disabled when perturbation is active because the test needs to be reframed in perturbed form.

```
z = 0; zsave = 0; steps = 0; limit = 2;
for n in 1..maxIter:
    z = z*z + c
    if |z|² > bailout²: return EXTERIOR(n)
    if |z - zsave|² < eps²: return INTERIOR     // cycle locked
    if ++steps == limit:
        zsave = z; steps = 0; limit *= 2
```

The two failure modes to guard against are **false positives** from slowly converging exterior orbits (mitigated by a short settle-down delay of ~64 iterations before enabling the check, and by the |z|>R escape test happening in the same loop) and **false negatives** from over-tight epsilon, which simply costs you the speedup. A typical epsilon in plain double is **1e-14 to 1e-16**, but with perturbation arithmetic the test must be on Z_n + Δz_n (the actual orbit), not on Δz_n alone, since Δz can grow large while the true orbit cycles. Claude Heiland-Allen's "biased" interior detection at mathr.co.uk turns the check on for tiles whose neighbors were interior and off for exterior-dominated tiles, recovering nearly the full speed of the no-check path in pure exterior regions. **Expected gain on the developer's 13–15 second minibrot tile: ~50–500× depending on the minibrot period**, because a p=200 cycle locks within roughly 2p–4p iterations, not 10000.

## Atom domains and Newton: the mathematically clean interior certificate

Periodicity is the operational fix; **atom domains plus Newton's method on the nucleus** is the principled one. The cardioid + period-2 bulb test the developer plans to add is the special case for p=1 and p=2. The general version works on any minibrot once you know its period.

An atom domain of period p is the set of c for which |F_c^p(0)| is the running minimum of |z_n| over n=1..p along the orbit from z=0. Each new minimum encountered while iterating the tile center is a **partial period**: a candidate p for an enclosing hyperbolic component. A more robust variant for a quadtree tile is the **box-period method** (Heiland-Allen's `m_d_box_period.c` in `mandelbrot-numerics`): iterate the four c-corners of the tile in lockstep, and at the iteration p where the four resulting z values wind around the origin once, p is the period of any hyperbolic nucleus inside the box. Ball arithmetic (Knighty) gives an analytically proven period using an interval enclosing the entire tile.

Given p, **Newton's method finds the nucleus c₀** as the unique solution of F_c^p(0) = 0, with the dual recurrence dc_{i+1} = 2·z_i·dc_i + 1 alongside z_{i+1} = z_i² + c. A few outer Newton iterations of p inner steps each, started from the partial-minimum location, converge quadratically. Once c₀ is known, the **size estimate** of the minibrot (Robert Munafo's "Mu Atom Sizes", mrob.com/pub/muency/muatomsizes.html) gives a disk of certified interior around c₀; if your tile lies inside it, **the entire tile fills with INTERIOR in O(1) per pixel**. For pixels outside the certified disk, a per-pixel Newton on z gives the attracting periodic point z* and the multiplier |F_c^p'(z*)| = |∏ 2·z_i|; the pixel is interior iff this is below 1. Each per-pixel test is O(p · few Newton steps) rather than O(maxIter), and **for typical deep-path minibrot periods of a few hundred this is a 5–20× reduction even before counting the tiles filled by size-estimate alone**.

Real implementations to study are Heiland-Allen's `mandelbrot-numerics` library (the cleanest small reference, GPL), `mandelbrot-perturbator` (the GTK explorer that combines partials + Newton + interior DEM end-to-end), Mandel Machine (uses Newton-found nuclei to center deep zooms), and FractalShark's Feature Finder (located at github.com/mattsaccount364/FractalShark, locates minibrots and refines their coordinates via Newton). Wolf Jung's Mandel program is the historical reference for Newton's method on Mandelbrot.

## Distance estimation supplements interior detection

The **exterior distance estimator** tracks the derivative dz/dc alongside z (dc_{n+1} = 2·z_n·dc_n + 1) and at escape returns d ≈ |z|·log|z|/|dz/dc|, an estimate of distance to the Mandelbrot boundary accurate to within a factor of 2 (Milnor; Inigo Quilez's iquilezles.org/articles/distancefractals/). The **interior distance estimator** (Heiland-Allen, mathr.co.uk/blog/2014-11-02_practical_interior_distance_rendering.html) needs the nucleus and period from the previous section, plus carrying both z-derivatives dz/dz, dz/dc and second derivatives, and produces d_int = (1 - |F'|²) / |d²F/(dzdc) + (d²F/dz²)·(dF/dc)/(1 - dF/dz)|. The practical upshot for this codebase is **flood filling**: once a pixel is known interior with distance d_int from the boundary larger than a pixel size, a disk of radius d_int/4 around it is also guaranteed interior, allowing wholesale skipping. Combined with the atom-domain certificate, this means most pixels in a deep minibrot interior require no iteration at all.

## Perturbation theory: how serious renderers do deep zooms

Doubles run out of precision around zoom 1e-15. Computing every pixel in MPFR is unacceptably slow, and the GPU cannot run MPFR at all. **Perturbation theory**, introduced by K.I. Martin in the 2013 "SuperFractalThing" paper (sft_maths.pdf), splits the problem: compute a single **high-precision reference orbit** Z_n once on the CPU in MPFR, then iterate every other pixel as a **low-precision delta** relative to that reference. The algebra is exact:

```
(Z + z)' = (Z + z)² + (C + c) = (Z² + C) + 2·Z·z + z² + c
         = Z_next + (2·Z·z + z² + c)
⇒ z_{n+1} = 2·Z_n·z_n + z_n² + c            // the perturbation formula
```

The catastrophic cancellation that would happen in (Z+z)² + (C+c) is performed symbolically before evaluation, so the large Z² + C term never actually contributes to z; the remaining terms 2Zz, z², and c are all of similar small magnitude and fit comfortably in hardware doubles. **This is what makes deep-zoom GPU rendering feasible**: the reference orbit is a one-time CPU cost (a table of M complex doubles, one per iteration), and every pixel becomes a fast hardware-double iteration on the GPU indexing into that table. The escape test stays exact: |Z_n + z_n|² > escape_radius². For the minibrot outlier problem specifically, perturbation also opens a clean route to interior detection on the perturbed orbit since rebasing-based cycle detection comes essentially for free (next section).

The technique has two failure modes that both have established cures. **Glitches** happen when the reference orbit passes close to zero (or close to another orbit), making |Z + z| smaller than |Z| and breaking the linearization. **Underflow** happens past zoom ~1e-300 when the per-pixel delta c itself becomes smaller than the double minimum exponent of 1e-308 even though the rendering is still correct in principle.

## BLA is the modern deep-zoom accelerator and the natural successor to series approximation

Series approximation (also K.I. Martin) expresses z_n as a polynomial in c, z_n = A_{1,n}·c + A_{2,n}·c² + ..., with coefficients that depend only on the reference orbit and satisfy the recurrence A_{k,n+1} = 2·Z_n·A_{k,n} + Σ_{j<k} A_{j,n}·A_{k-j,n}. After computing these once, every pixel can initialize its delta-iteration loop at iteration N for free where N is the largest count for which the truncated series stays accurate across the image. This is what Kalles Fraktaler 2, Mandel Machine, and Ultra Fractal use.

**Bivariate Linear Approximation (BLA)**, due to Zhuoran in 2021 on fractalforums.org (topic 4360), has largely replaced SA in modern code. The idea: when |z_n| is small enough that the z² term in the perturbation formula drops below the noise floor, the iteration becomes a linear map z_{n+1} = 2·Z_n·z_n + c. Linear maps compose into linear maps, so many consecutive iterations can be collapsed into a single linear map z_{n+l} ≈ A·z_n + B·c with a validity radius R such that the approximation holds whenever |z| < R. A precomputed table of (A, B, R) for many starting iterations and many skip lengths l lets each pixel, at each iteration, look up the largest valid skip and jump ahead.

The single-step BLA is just A₁ = 2·Z_n, B₁ = 1, with **validity radius R = (ε·|Z_{n+1}| − |c|)/|2·Z_n|** for some small ε (≈ 2·10⁻⁵³ for double, sometimes auto-tuned much smaller per location). Two adjacent BLAs merge with A_z = A_y·A_x, B_z = A_y·B_x + B_y, R_z = min(R_x, (R_y - |B_x|·|c|)/|A_x|). Building the table as a binary tree by pairwise merging yields ~2M entries for M reference iterations, and Zhuoran's "merge and cull" trick drops levels 1 and 2 to roughly halve memory at almost no performance cost. Per-pixel lookup descends from the largest level to the smallest, taking the first valid skip — O(log M) lookups per iteration. **Reported speedups are 10–100× over plain perturbation**: Fraktaler-3 renders a 5e-433 minibrot in ~2 minutes versus ~20 minutes in Kalles Fraktaler's NanoMB1+SA combination.

The major implementations are **Fraktaler-3** (Claude Heiland-Allen, AGPL3, OpenCL — the canonical open-source BLA implementation, at code.mathr.co.uk/fraktaler-3.git), **Imagina** (Zhuoran's own renderer, github.com/5E-324/Imagina, the original BLA + rebasing implementation), **FractalShark** (Matt Saccone, GPL3, CUDA — the clearest example of mapping BLA onto NVIDIA hardware with a custom "2×32 + shared-exponent" type that beats emulated double on consumer GeForce cards), and **fractalshades** (Geoffroy Billotey, MIT, Python). Phil Thompson's blog post at philthompson.me/2023/Faster-Mandelbrot-Set-Rendering-with-BLA-Bivariate-Linear-Approximation.html is the best tutorial-quality writeup with full formulas and the auto-ε binary search heuristic. **For a new implementation, skip SA entirely and go straight to BLA** — it generalizes to non-analytic formulas like Burning Ship via a 2×2-matrix variant, which SA does not.

## Rebasing has replaced glitch-detect-and-recompute

The classical glitch fix from Pauldelbrot (fractalforums.com, 2014) detects a pixel as glitched when **|Z_n + z_n|² < G·|Z_n|²** for a tolerance G typically in [1e-2, 1e-8], then picks a new reference inside the largest glitched region, recomputes that reference in MPFR, and re-iterates the flagged pixels. This works, is fully implemented in Kalles Fraktaler 2 and DeepDrill, but tuning G is fragile — too small misses glitches, too large makes everything look glitched and runs orders of magnitude slower.

**Zhuoran's rebasing (fractalforums.org/topic=4360, 2021)** is the elegant modern replacement: when |Z_m + z_n| < |z_n| during iteration — that is, when the orbit returns near the critical point at z=0 — reset the reference iteration index m to 0 and continue. Since the reference orbit also started from z=0, the pixel's current z is now exactly the right low-precision representation of where it is, and iteration resumes cleanly. **Only one reference orbit is needed per image** for Mandelbrot, glitches are *avoided* rather than detected and patched, and the runtime check is free.

```
m = 0; dz = 0
for n in 1..maxIter:
    dz = 2*Reference[m]*dz + dz*dz + dc
    m += 1
    z = Reference[m] + dz
    if |z|² > bailout²: return n
    if |z| < |dz| or m == M_ref: dz = z; m = 0   // rebase
```

Rebasing is now the standard in Fraktaler-3, Imagina, FractalShark, and FractalZoomer. Kalles Fraktaler 2 still uses Pauldelbrot detection because rebasing was developed after KF2's main development ended. The rebasing test is also exactly the condition under which BLA validity must be re-checked, so the two integrate cleanly. **For a new implementation, skip Pauldelbrot detection and go straight to rebasing** — it is simpler, faster, and doesn't need a magic threshold.

## Extended precision: where to spend bits

The precision stack is well-established. For the reference orbit only — computed once per image on the CPU — use **MPFR** (or MPIR on Windows for AVX2-tuned big multiply) at precision roughly 53 + log₂(zoom) + 16 bits. The reference orbit is O(maxIter · M(prec)) and parallelizes only via internal SIMD; for maxIter ≈ 10000 and zoom 1e1000 this is seconds to minutes, paid once. Convert each Z_n to the per-pixel type and upload to the GPU.

For the **per-pixel delta** the choice depends on depth. Up to zoom 1e300, hardware double suffices and is what gets you the 50–100× over MPFR-per-pixel that perturbation buys. Past 1e300, double-precision underflow during intermediate iterations becomes the binding constraint even though the final answer is fine. The two cures are **floatexp** (a double mantissa with a separate int32 or int64 exponent, with software-emulated arithmetic about 3–5× slower than hardware double — Heiland-Allen's `floatexp.h` in mandelbrot-perturbator is the reference implementation) and **rescaled doubles** (substitute z = S·w, c = S·d, cancel a factor of S, drop the +S·w² term when S underflows; faster than floatexp for "well-behaved" locations, slower for "deep needle" locations where every iteration takes a full-range path).

Bailey/Hida/Li's **double-double (~106 bits mantissa) and quad-double (~212 bits)** from the QD library are useful at intermediate depths where you want extra mantissa precision without touching the exponent range — Fraktaler-3 ports portions of QD to GLSL for 49-bit GPU types, and FractalShark's "2×32 + shared exponent" gets ~48 mantissa bits at much higher throughput than emulated double on consumer GPUs where fp64 is throttled to 1/32. Reference orbit memory becomes a real constraint at maxIter > a few million; **Zhuoran's reference compression** (only key iterations near periodic returns are stored, intermediate values recomputed by short re-iterations) is implemented in Imagina and FractalShark and becomes essential past zoom ~1e1000 with high-period minibrots.

## SIMD is a near-free 4–8× multiplier on the bottleneck

The inner loop is two complex multiplies and an add. On AVX2 with doubles, four pixels iterate per instruction; on AVX-512, eight. The classic divergence problem — escaped lanes idle while non-escaped lanes keep iterating — has two mitigations: **mask-and-wait** (the simplest, used by Wojciech Muła's PTEST-based SSE4.1 implementation, achieving ~4.5× over scalar) and **lane refilling** (when a lane escapes, gather a fresh pixel from the work queue into that lane and reset its counter; recovers efficiency in divergent boundary regions, ~3–4× sustained in hard cases). Chris Wellons' SSE2/AVX/NEON implementation at nullprogram.com/blog/2015/07/10/ reports 5.6× in single precision; Muła's implementation is at github.com/WojciechMula/toys/tree/master/sse4-mandelbrot.

The crucial point for the developer's minibrot bottleneck: **every lane in the minibrot tile iterates to maxIter together with zero divergence**. There is no idle-lane cost, no refill cost, and SIMD delivers a near-theoretical 4× (AVX2) or 8× (AVX-512) reduction on the worst-case tile *before* any algorithmic change. Combined with periodicity detection from §3, this stacks multiplicatively — 4× × 50× = 200× on a tile that today costs 13 seconds. ISPC (Intel SPMD compiler) is the recommended tool for writing this without hand-intrinsics; Aiden Munro's github.com/aidenfmunro/MandelbrotSet is a clean reference for the SoA pixel-buffer layout that makes SIMD efficient.

## Async CUDA streams remain the immediate, highest-confidence GPU win

The developer's profiling already identified this: ~13 ms per region of recoverable GPU idle from synchronous cudaMemcpy blocking on the kernel rather than on the actual ~15 μs PCIe transfer. The fix is the canonical NVIDIA pattern (Mark Harris's "How to Overlap Data Transfers in CUDA C/C++" on the NVIDIA developer blog): replace `cudaMemcpy` with `cudaMemcpyAsync` on a non-default stream, allocate host buffers with `cudaMallocHost` (pinned — pageable memory silently degrades async to sync, which is the #1 pitfall), keep 2–3 buffer slots in rotation, and use `cudaStreamSynchronize` per slot instead of `cudaDeviceSynchronize`. With three rotating slots, one region's D→H drains while the next region's kernel runs and the third's input is being packed. NVIDIA's example sees end-to-end time drop from 7.10 ms to 3.97 ms (1.8×) when transfer ≈ compute; in this codebase where transfer is tiny but kernel is large, the win is **freeing the GPU driver thread to issue more work** rather than hiding transfer behind compute. Recovered idle is exactly the developer's measured 13 ms × number of GPU regions per frame. **Implementation effort is one weekend**, mostly mechanical changes to buffer ownership.

The complementary CUDA Graphs API (Ekelund/Markidis/Peng, arXiv:2501.09398) captures the per-region memcpy+kernel+memcpy sequence once and replays it with `cudaGraphLaunch`, reducing per-node launch overhead by ~3–5×. Worthwhile for offline batch rendering across 100 zoom frames where the per-frame structure is similar.

## Priority-queue scheduling helps; work-stealing does not

Routing the biggest, most-expensive regions to the GPU first is a textbook **Longest Processing Time First (LPT)** heuristic (Graham 1969, with a worst-case bound of 4/3 − 1/(3m) versus optimal makespan on m identical machines). For this codebase's 1 GPU + 11 CPU = heterogeneous 12 workers, the cleanest implementation is a priority queue keyed by predicted cost, with a routing rule that pushes high-cost regions into a GPU-only sub-queue and lets CPU threads work the rest. **Cost estimation** can be `area × (1 − exterior_fraction) × maxIter` using the existing border sample, refined by 4–16 random interior probes if any of them reach a large fraction of maxIter. This is low-complexity and gives a solid 1.2–1.5× on mixed frames.

The honest caveat the developer's reports already noted: priority queueing alone does not fix the minibrot frame, because the binding region still goes to *some* worker. Even on the GPU, a 960×540 interior tile at 40× CPU speed still takes ~325 ms — better than 13 s but still potentially the longest single job in the frame and therefore the makespan bottleneck. Priority queueing is a complement to, not a substitute for, the interior-certificate and GPU-subdivision approaches.

**Work-stealing schedulers (Cilk-style per-thread deques with random victim selection) are explicitly not worth implementing in this codebase.** Blumofe & Leiserson's algorithm wins when queue contention is high or task sizes are very small and irregular. The codebase already reports <2–4% thread spread with a single shared deque, which proves contention is not a bottleneck. Work-stealing would replace a working centralized queue with a more complex per-thread deque system to recover under 1%, and the residual imbalance is tail-effect from a single thread stuck on a giant region — which work-stealing cannot redistribute, since the work is already in some thread's deque and being executed.

## GPU-side adaptive subdivision is the real upper bound

The biggest unexplored throughput win is moving the quadtree subdivision itself onto the GPU. Today, when a region needs subdividing, the host samples its border, decides to split, and pushes four sub-regions back into the deque — each step crossing the PCIe + synchronization boundary. **Andy Adinets' Mariani-Silver Mandelbrot CDP example** (developer.nvidia.com/blog/introduction-cuda-dynamic-parallelism/) implements the same algorithm with one initial kernel launch; each block evaluates its tile's border, flood-fills if uniform, recurses on sub-tiles via `cudaLaunchKernel` from device code.

CUDA Dynamic Parallelism (CDP) has caveats — CDP2 dropped device-side `cudaDeviceSynchronize`, naïve CDP suffers from per-launch overhead, and the Losor evaluation finds CDP only beats a standard kernel at >12k² images. The recommended modern alternative is **Adaptive Serial Kernels (ASK)** from Quezada, Navarro et al. (arXiv:2206.02255 — they use Mandelbrot as their case study), where the host launches a *sequence* of kernels, each reading work items from a device buffer and writing new sub-items to another. They report **ASK up to 60% faster than CDP and 12× faster than a flat exhaustive kernel** on Mandelbrot, with optimal subdivision parameters {initial grid ~32, recurrent ~2, stop size ~32}. A reference implementation lives at github.com/temporal-hpc/GPU-dynamic-parallelism. For 2026 implementations this is the better choice than classical CDP.

The **wavefront iteration-stream pattern** from Laine/Karras/Aila's "Megakernels Considered Harmful" (HPG 2013) is the orthogonal divergence fix: instead of letting warps run iteration loops with mixed escape times, keep a pool of alive pixels in a device buffer, run N iterations per kernel pass, compact escaped pixels out, refill from the work queue. Path-tracing developers treat this as the canonical divergence cure; for deep-zoom Mandelbrot where some pixels take 10⁶ iterations and others escape immediately, this potentially adds another 1.5–3× on top of GPU subdivision. The classical persistent-threads alternative (Tzeng/Patney/Owens, HPG 2010) achieves similar effects with one long-running kernel pulling work from a GPU-side queue via `atomicAdd`, but is much harder to get right and rarely beats ASK in practice.

## A prioritized roadmap calibrated to your codebase

The order matters because the wins stack, the dependencies run one way, and the developer effort is unevenly distributed. The recommended sequence is:

1. **AVX2 SIMD on the existing inner loop** is the first move because it costs nothing algorithmically, multiplies the worst-case tile by 4× immediately even before any other change, and is easiest via ISPC. Expect a 3–4× wall-clock reduction across the board with little code disruption.
2. **Brent's periodicity detection** stacks directly on top of SIMD and is the actual cure for the minibrot outliers. With a 64-iteration settle delay, epsilon 1e-14 in double, and the test gated on tiles whose neighbors were interior, expect another 10–100× on the bottleneck tiles. Combined with SIMD this is 40–400× on the worst-case region with a few hundred lines of code.
3. **Asynchronous CUDA streams with pinned host buffers and triple-buffering** recovers the 13 ms × G/frame GPU idle the developer measured. About a weekend of mechanical work, ~1.5–2× on the GPU portion of the frame.
4. **Atom-domain + Newton-found nucleus + multiplier-based interior certificate** replaces ad-hoc periodicity with the principled per-tile certificate the developer's roadmap was already converging on. Use box-period from the four corners, Newton on F^p(0)=0 for the nucleus, and the size-estimate disk to flood-fill certified tiles in O(1). This subsumes the cardioid + period-2 bulb test and generalizes to every minibrot.
5. **Priority-queue scheduling with cost estimation by interior fraction** routes expensive regions to the GPU first. Low complexity, 1.2–1.5× on mixed frames, sets up the next step.
6. **Perturbation theory + Zhuoran rebasing in double precision** to unlock zoom past 1e-15. Compute the reference orbit in MPFR on the CPU, upload the table to the GPU, run the perturbed iteration in hardware doubles per pixel with the inline rebasing check. Skip Pauldelbrot detection and series approximation entirely; both are obsolete in modern code.
7. **BLA tables on top of perturbation** for the 10–100× per-pixel skip. Build the table on the CPU by pairwise binary-tree merging starting from iteration 1, drop levels 1–2 per Zhuoran's merge-and-cull, ship the (A, B, R) arrays per level to the GPU, and have each kernel thread do descending-level lookups in its iteration loop.
8. **floatexp (and optionally rescaled doubles) for the per-pixel deltas** to push depth past 1e-300. Software-emulated arithmetic, ~3–5× slower than hardware double, but mandatory beyond double underflow. Heiland-Allen's `floatexp.h` is the reference.
9. **GPU-side adaptive subdivision via ASK** is the highest-ceiling remaining win for throughput on minibrot frames once perturbation is in place. Implementation is real work (1–2 weeks for a clean version), but the speedup ceiling is the largest of any remaining technique.
10. **Wavefront iteration-stream pixel pool** on the GPU for divergence-heavy deep-zoom frames, layered on top of ASK. Adds another 1.5–3× when iteration counts vary wildly per pixel.

The two genuinely game-changing items are #2 (periodicity detection) and the perturbation/BLA stack #6+#7+#8. Periodicity directly retires the minibrot-outlier bottleneck the developer's profiling has pinned down. Perturbation+BLA+floatexp is what makes "arbitrary deep zoom" actually arbitrary; without it the renderer is permanently bounded at zoom 1e-15 regardless of how clever the scheduler gets. Async streams (#3) and SIMD (#1) are the cheap throughput multipliers that should ship first because they require no algorithmic risk and stack with everything else. Work-stealing should be skipped; the current load-balance numbers prove it would not pay off, and the developer's time is better spent on the interior-certificate and deep-zoom work that addresses real bottlenecks rather than imaginary ones.
