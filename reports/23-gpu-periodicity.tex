\begin{abstract}
This report ports the exact Brent periodicity check of
\texttt{21-periodicity-check.tex} from the CPU \texttt{diverge()} to the CUDA
kernel --- the variant report~21 explicitly deferred with ``expect a loss.''
Measured on the production spec (100 frames, $1920\times1080$,
\texttt{diffT}\,=\,0.1, \texttt{save}\,=\,1, 3 alternating reps with a CPU12
control, on AC): the prediction is \textbf{falsified}. Mix-controlled kernel
time over the identical 5{,}608 regions falls $66.2 \to 28.3$~s
(\textbf{2.34}$\times$), GPU-only wall falls $78.2 \to 40.1$~s ($-48.8\%$),
and the production hybrid falls $\mathbf{29.83 \to 23.72}$~\textbf{s}
($\mathbf{-20.5\%}$) --- the best wall this project has recorded --- while
all 100 output frames stay \textbf{byte-identical}. The error in the
prediction was its cost model: a warp does not pay $32\times$\texttt{MAXITER}
when lanes diverge at an early exit; it pays the \emph{slowest lane's}
detection latency, and spatially adjacent lanes detect their cycles at
correlated times. The hybrid self-rebalances a third time (GPU share
$18 \to 54\%$ of regions), the CPU pool stays the binding resource at 20\%
less cost, and the GPU thread's own cost structure inverts: kernel occupancy
drops from 85\% to 45\% of the wall, promoting host-side commit --- report
20's \texttt{scanLine} lever --- to the next target.
\end{abstract}

\section{Motivation: the last executor still grinding interiors}
\label{sec:motivation}

\texttt{binary-v6-periodicity} (report~21) eliminated the interior grind on
the CPU only: a Brent-style cycle detector returns \texttt{MAXITER} as soon
as the orbit state exactly revisits a saved state, cutting CPU12 by
$-78.6\%$ and the hybrid by $-46.3\%$ with byte-identical output. The GPU
kernel was deliberately left untouched, on the strength of two kernel-level
measurements from report~16: interior regions run at exactly 100.0\% warp
execution efficiency, and the kernel is FP64-bound at 85.6\% of peak.
Report~21's recommendation~6 spelled the prediction out: \emph{``A/B a
GPU-side periodicity variant separately --- expect a loss \ldots\ per-pixel
early-out would introduce divergence into that kernel and add FP64 compare
work to the bottleneck pipe.''}

Two developments made that prediction worth testing rather than trusting.
First, the repricing analysis of report~22 (branch \texttt{feat/igpu-opencl})
showed that v6 devalued every executor whose niche was interior content ---
the dGPU's marginal worth fell to $\sim$3.4 CPU-worker equivalents
(\S\ref{sec:repricing}) precisely because affinity feeds it the big interior
regions that CPUs now skip most of. The dGPU was left as the \emph{only}
executor still grinding every interior pixel to \texttt{MAXITER}: on the v6
hybrid, the GPU thread spends 85\% of the entire wall inside kernels
(Fig.~\ref{fig:lane}), most of it proving what a cycle detector knows after
$\sim$900 iterations (report~21's measured detection latency on this
content).

Second, the prediction's cost model does not survive contact with how warps
actually execute an early exit. A lane that detects a cycle and returns does
not stall its warp; it sits \emph{masked} while the remaining lanes iterate.
The warp's cost is therefore not $32\times$\texttt{MAXITER}-or-nothing --- it
is the \emph{maximum} of its 32 lanes' detection latencies. The 32 lanes of a
$16\times16$-block warp are spatially contiguous pixels of the same interior
region, attracted to the same cycle, so their latencies are strongly
correlated and the maximum is a small multiple of the mean. Against
\texttt{MAXITER}\,=\,10{,}000 on the deep frames, even a $3\times$ inflation
of the $\sim$900-iteration mean latency wins by an order of magnitude. The
real costs are the two extra FP64 compares per iteration on \emph{every}
lane ($\sim$25--30\% on the $\sim$7-FLOP recurrence, and on the bottleneck
pipe, exactly as report~21 said) and the forfeited efficiency metric. Whether
the latency win covers those costs is a measurement, not a derivation ---
hence this experiment.

\section{The change: the CPU detector, verbatim, in the kernel}
\label{sec:change}

Commit \texttt{2abc909} adds the same anchor-and-compare block that
\texttt{fa6d8ac} added to \texttt{MandelRegion::diverge()}, with the same
doubling refresh schedule (anchor starts at $z_0$, refreshes at iterations
8, 16, 32, \ldots), to the device function in \texttt{kernel.cu}
(comment abridged here; the full exactness argument is in the source):

\begin{sourcecode}{cpp}{kernel.cu --- the device \texttt{diverge()} with the Brent check (commit \texttt{2abc909}).}
__device__ int diverge(double cx, double cy, int MAXITER) {
  int iter = 0;
  double vx = cx, vy = cy, tx, ty;
  // Brent-style periodicity check, mirroring the CPU diverge():
  // an exactly-repeating orbit can never escape, so returning
  // MAXITER on exact FP revisit is exact w.r.t. this kernel's own
  // (FMA-contracted) arithmetic. A detecting lane exits and sits
  // masked; the warp's cost drops to its slowest lane's latency.
  double sx = vx, sy = vy;
  int nextSave = 8;
  while (iter < MAXITER && (vx * vx + vy * vy) < 4) {
    tx = vx * vx - vy * vy + cx;
    ty = 2 * vx * vy + cy;
    vx = tx;
    vy = ty;
    iter++;
    if (vx == sx && vy == sy)
      return MAXITER;
    if (iter == nextSave) {
      sx = vx;
      sy = vy;
      nextSave *= 2;
    }
  }
  return iter;
}
\end{sourcecode}

The exactness argument is per-device: the iteration map as compiled for
\emph{this} GPU is deterministic, so an exact state revisit proves the orbit
cycles forever and the plain loop would have ground to \texttt{MAXITER} and
returned the same value; an escaping orbit never exactly repeats, so
exterior pixels are untouched. The one new hazard relative to the CPU case
is codegen: the guarantee is against the kernel's own FMA-contracted
arithmetic, so output identity additionally requires that \texttt{nvcc}
contract the (textually unchanged) iteration statements identically in both
builds. That is an empirical question, and \S\ref{sec:identity} answers it:
it did.

No other code changes. In particular the BlockingSync probe of report~22
lives only on \texttt{feat/igpu-opencl} --- one lever per branch, so this A/B
measures the periodicity check and nothing else.

\section{Methodology}
\label{sec:method}

\textbf{Hardware/workload.} i7-9750H (6C/12T) + GTX 1660 Ti Max-Q,
production \texttt{spec.in} (100 frames, $1920\times1080$, deep zoom to
\texttt{maxIter} 10{,}000), \texttt{diffT}\,=\,0.1,
\texttt{pixT}\,=\,32{,}768, \texttt{quiet}\,=\,1, \texttt{save}\,=\,1 ---
the Fig-11.15 methodology of experiments 21 and 22. AC power.

\textbf{Configurations.} \texttt{experiments/23-gpu-periodicity/ab.sh}: three
configs $\times$ two binaries $\times$ three reps, binaries alternated
back-to-back inside each config so thermal drift hits both sides equally.
\emph{gpuonly} (1 thread, GPU enabled) isolates the kernel change --- every
region of every class goes through the modified kernel. \emph{hybrid}
(12 threads) is the production config. \emph{cpu12} (12 threads, GPU off) is
a control --- the CPU code is byte-identical between the binaries, so any
delta it shows is environment, not code --- and the same-batch anchor for
the worker-equivalence arithmetic of \S\ref{sec:repricing}. The baseline
binary was built from \texttt{main} (\texttt{c4744c0}) in a clean worktree.

\paragraph{Calibration honesty.} The battery was charging during the sweep
(added thermal load) and the sweep started on cold silicon. The control
caught both: its per-rep delta reads $+6.4\% \to +1.2\% \to +0.07\%$ across
reps 1--3 --- while its \emph{pool compute} is 408.6~s in both binaries to
within 0.1~s --- so the batch reaches thermal steady state during rep~2 and
the control then converges to the wash it must be. Because \texttt{base}
runs first in every pair, rep~1's cold bias favours the baseline, i.e.\
3-rep means \emph{understate} the improvement. This report therefore quotes
\textbf{steady-state numbers (reps 2--3)} with 3-rep means alongside.
Repeatability at steady state is excellent: the two \texttt{gp} hybrid reps
agree to 1.4~ms. An earlier same-day non-charging batch
(\texttt{logs/smoke\_hybrid.out}) ran $\sim$1.5~s faster in absolute terms
at the same relative delta (hybrid $27.93 \to 22.44$~s, $-19.7\%$) ---
consistent with the $\sim$10\% thermal-history band documented in
report~22.

\section{Analysis}
\label{sec:analysis}

\subsection{The ``expect a loss'' prediction is falsified, and the error is
instructive}

Mix-controlled kernel time fell 2.34$\times$ (Table~\ref{tab:gpuonly});
nothing about the result is marginal. The prediction failed because it
priced the check with the wrong cost model. Report~16's 100.0\% warp
execution efficiency on interior regions was read as an asset the check
would destroy --- but on interior content that metric \emph{measured the
waste itself}: every lane of every warp executing all 10{,}000 iterations
in lockstep is precisely what ``100\% efficiency, zero early exits'' means.
The check trades metric efficiency for eliminated work: a lane that detects
its cycle at iteration $\sim$900 sits masked for the remainder of the warp's
execution, and masked-idle costs nothing, whereas the ``efficient''
alternative was 9{,}000 further iterations of provably useless FP64. The
warp pays the maximum of its lanes' detection latencies, and because a
warp's 32 lanes are spatially contiguous pixels attracted to the same cycle,
that maximum is a small multiple of the mean --- not \texttt{MAXITER}. The
implication generalises: warp execution efficiency is a throughput metric,
valid only when the work being kept coherent is \emph{useful}; reports
16/17's 100\% baselines are hereby stale by design
(recommendation~\ref{rec:ncu}).

\subsection{Why 2.34$\times$ and not the CPU's 4.8$\times$: the warp pays
the max, and every lane pays the compares}

The same check cut the CPU pool's compute 4.77$\times$ on this workload
($1{,}917 \to 402$~s, report~21) but the kernel only 2.34$\times$. The gap
has two mechanisms, both structural. First, granularity: a CPU thread exits
each pixel at that pixel's own detection latency, while a warp runs every
pixel to the \emph{slowest} of its 32 lanes --- the max of 32 correlated
latencies still exceeds their mean. Second, overhead incidence: the two
FP64 compares are paid per iteration by \emph{every} lane --- including the
escaping boundary pixels that never benefit --- on a kernel that report~16
measured at 85.6\% of the FP64 pipe. The check makes the per-iteration
recurrence $\sim$25--30\% more expensive and then wins anyway, because it
deletes 80--90\% of the iterations on interior content. Both mechanisms
are intrinsic to SIMT early-exit; 2.34$\times$ is what ``expect a loss''
actually looks like when measured.

\subsection{The hybrid self-rebalances a third time; the pool binds at 20\%
less cost}

The wall win is a clean two-step transfer, and both steps are in
Table~\ref{tab:hybrid}. Step one: the kernel gets 7$\times$ cheaper per
region ($25.10 \to 3.56$~ms, partly mix --- the GPU's marginal regions are
now smaller), so the min-max affinity queue of report~12 automatically
hands the GPU 3,006 regions instead of 1,008 (Fig.~\ref{fig:share}). This
is the third self-rebalance in the series: v5's GPU held $\sim$61\% of the
queue, v6's CPU-side check pulled it down to $\sim$18--24\% (the CPUs got
fast, the GPU didn't), and v7 restores $\sim$54\% now that both sides skip
the grind. Step two: the 2{,}000 migrated regions carry 62~s of CPU-pool
compute ($311.6 \to 249.6$~s, $-19.9\%$), and since the busiest CPU worker
tracks the wall in both binaries (within 3\%, Fig.~\ref{fig:pool}), the
pool relief converts one-for-one into wall: $-20.5\%$. The load balance
itself never degrades --- the pool's 11 workers stay within 3\% of each
other --- which is the report-13 result surviving its third redistribution.

\subsection{The GPU lane's cost structure inverts: the next lever is host-side
commit, and it now transfers to wall}

In the v6 hybrid the GPU thread spent 84.9\% of the wall inside kernels;
in v7 that is 45.1\%, and the lane's majority cost is now its own host
side: 12.77~s of per-pixel \texttt{setPixel} commit, stencil sampling and
queue work against 10.69~s of kernel (Fig.~\ref{fig:lane}). Real D$\to$H
transfer stays negligible (0.26~s total, $\sim$86~\textmu s/region ---
report~15's finding, re-confirmed). Two facts make this actionable. First,
the host cost is per-region invariant across binaries ($4.4$ and
$4.3$~ms/region) --- it is commit work proportional to pixels, untouched by
the check, exactly as expected. Second, the lane and the pool now finish
together (both $\approx$ wall), so freeing lane time is no longer cosmetic:
a cheaper commit lets the GPU pull further regions off a pool that is still
the binding resource. Report~20's \texttt{scanLine} row-copy lever --- replace
the per-pixel \texttt{QImage::setPixel} loop with row-wise writes --- was
priced on the iGPU branch as a lane optimisation; on v7 it graduates to the
top of the wall-relevant list, with $\sim$12.8 of 23.7~s as its theatre of
operations (bounded, of course, by how much of that the pool can absorb).

\subsection{Repricing, round three: the dGPU doubles its worker-equivalence}
\label{sec:repricing}

With the same-batch CPU12 anchor (35.78~s steady state), the
worker-equivalence of report~22, $k = 12\,(t_{\mathrm{CPU12}}/t_{\mathrm{config}}) - 11$,
prices the dGPU at $k = 3.4$ CPU workers under v6 and $\mathbf{k = 7.1}$
under v7 --- the check more than doubles the marginal value of the device
it runs on, the mirror image of v6 halving it. The consequence for the
parked \texttt{feat/igpu-opencl} branch is that its wash result (report~22)
is stale in \emph{both} directions: the iGPU's OpenCL kernel can take the
same 15-line check (raising its raw speed), while the twice-repriced dGPU
now clears the interior content the iGPU was queuing for (lowering the
relief it can sell to the still-binding pool). Which force wins is a
measurement (recommendation~\ref{rec:igpu}), not a derivation --- report~21
teaches what trusting the derivation costs.

\section{Recommendations}
\label{sec:recommendations}

\begin{enumerate}
  \item \textbf{Merge \texttt{feat/gpu-periodicity} to \texttt{main}; tag
        \texttt{binary-v7-gpu-periodicity}.} Identity gate 100/100, control
        clean at steady state ($+0.07\%$ in rep 3), hybrid $-20.5\%$
        (23.72~s, best recorded), GPU-only $-48.8\%$. Known trade-off: two
        FP64 compares per iteration are pure overhead on exterior-only
        content --- bounded at the $\sim$25--30\% per-iteration figure,
        and report~18 showed uniform-exterior is this workload's cheap case;
        accept it, as report~21 did for the CPU.
  \item \textbf{Cheapen the GPU thread's commit path
        (\texttt{scanLine} row copies instead of per-pixel
        \texttt{setPixel}).} The lane's host side is now its majority cost
        (12.77~s vs 10.69~s kernel) and the pool it feeds still binds the
        wall, so lane time freed here converts to wall until the two
        re-equalise. Expected shape: same two-step transfer as
        \S\ref{sec:analysis}.3. Cost: a contained change in
        \texttt{mandelregion.cpp}'s GPU commit loop; A/B on its own branch.
  \item \label{rec:ncu} \textbf{Re-baseline the ncu warp metrics on v7.}
        Reports 16/17 pinned interior kernels at 100.0\% warp efficiency;
        the check spends that metric deliberately. One capture per the
        \texttt{experiments/16}/\texttt{17} scripts, so the number on
        record explains itself and nobody ``fixes'' the kernel back to
        efficient waste. Cost: one sudo capture session.
  \item \label{rec:igpu} \textbf{Re-price the iGPU on top of v7, after
        porting the check to \texttt{oclkernel.cpp}.} Same 15-line mirror,
        same identity gate (per-device exactness holds for the OpenCL
        compiler's arithmetic too --- byte-identity is only expected against
        its own baseline, as report~20 documented). Report~22's wash is
        stale in both directions (\S\ref{sec:repricing}); one
        experiment-22-shaped sweep decides it.
  \item \textbf{Keep the cold-start discipline in future sweeps.} The
        control caught a $+6.4\%$ rep-1 artefact that alternation alone did
        not remove (base runs first in every pair). Discard-first-rep ---
        or a warm-up run --- plus a null-config control should be standard
        for every future A/B on this machine; \texttt{ab.sh} already
        refuses to run off AC.
\end{enumerate}

\section{Conclusion}

The GPU-side periodicity check does what report~21's recommendation said to
measure and the opposite of what it said to expect: mix-controlled kernel
time $2.34\times$ ($66.2 \to 28.3$~s over the identical 5{,}608 regions),
GPU-only wall $-48.8\%$, production hybrid $\mathbf{29.83 \to 23.72}$~s
($\mathbf{-20.5\%}$, steady state, cold-start rep excluded by a CPU12
control that converges to $+0.07\%$) --- the best wall this project has
recorded, with all 100 frames byte-identical and the 5{,}608-leaf
decomposition unchanged. The prediction failed on its cost model: a warp
pays its slowest lane's detection latency, not lockstep \texttt{MAXITER},
and 100\% warp execution efficiency on interior content was a measurement
of the waste, not of health. The hybrid self-rebalances a third time (GPU
share $18 \to 54\%$), the CPU pool remains the binding resource at 20\%
less cost, and the GPU thread's cost structure inverts --- kernel occupancy
$85 \to 45\%$ of wall --- promoting host-side commit to the next lever.
Open items: the 3-rep confirmation is done, so the merge is unblocked; the
\texttt{scanLine} commit path, the ncu re-baseline, and the third iGPU
repricing are queued behind it.
