\begin{abstract}
\texttt{WorkQueue}'s FIFO \texttt{deque} is replaced with a largest-first
\texttt{priority\_queue} (the existing \texttt{MandelRegion::Compare}, corrected
to order by descending pixel count), in the hope --- per
\texttt{08-region-metrics-viz.tex}'s next-step~4 --- of routing big regions, and
in particular the $\sim$14~s interior outlier, to the fast GPU thread.
Measured against the FIFO baseline on 100~frames at $1920\times1080$
(i7-9750H + GTX~1660~Ti~Max-Q, 12~threads, \texttt{diffT}\,=\,0.1,
\texttt{save}\,=\,0), the change is \textbf{wall-neutral}: 51.09~s vs.\ 51.37~s
over 3~reps each ($-0.55\%$, within noise). The Nsight Systems breakdown shows
why nothing moves: \textbf{the GPU is equally utilised either way}
($\approx$46.5~s busy, $\sim$91\%), the \textbf{kernel max stays at
$\approx$139~ms} in both --- i.e.\ \textbf{the outlier is never routed to the
GPU}; it still lands on a CPU thread (worst region 13.3~$\to$~14.4~s) --- and
total compute is conserved ($\approx$610~s). With eleven CPU workers versus one
GPU thread, a \emph{shared} largest-first queue hands big regions to whichever
of the twelve extracts next, which is a CPU thread $\sim$11:1; and with the load
already balanced under 2\,\% (report~09), reordering a work-conserving,
throughput-bound schedule has nothing to gain. The report also documents a
methodology near-miss: a battery-power run \emph{falsely} showed a 2$\times$
``regression'' because the GPU was power-capped to 30~W; an AC-power sanity gate
caught it before any conclusion was drawn.
\end{abstract}

% -----------------------------------------------------------------------
\section{Setup and the Change Measured}

\paragraph{Workload and machine.} Identical to reports~09: same \texttt{spec.in}
(100~frames, $1920\times1080$, zoom to \texttt{maxIterations}\,=\,10{,}000),
i7-9750H (6c/12t) + GTX~1660~Ti~Max-Q, 12~threads (1~GPU~+~11~CPU),
\texttt{diffThreshold}\,=\,0.1, \texttt{pixelSizeThresh}\,=\,32{,}768,
\texttt{save}\,=\,0. \textbf{On AC power} (see Section~5).

\paragraph{The change.} \texttt{WorkQueue} held a FIFO \texttt{deque} (push back,
pop front). It is replaced by a \texttt{std::priority\_queue} ordered by pixel
count so \texttt{extract()} returns the largest pending region.

\begin{sourcecode}{cpp}{workqueue.h / mandelregion.h --- largest-first queue.}
// workqueue.h
priority_queue<MandelRegion *, vector<MandelRegion *>, MandelRegion::Compare> queue;

// mandelregion.h -- Compare corrected to largest-first.
// priority_queue pops the element that is "greatest" under operator()(a,b)=="a
// before b"; returning a<b makes top() the region with the MOST pixels.
struct Compare {
  bool operator () (const MandelRegion *a, const MandelRegion *b) {
    return a->pixelsX * a->pixelsY < b->pixelsX * b->pixelsY;   // was '>': smallest-first
  }
};
\end{sourcecode}

The existing \texttt{Compare} (dead code until now) returned \texttt{>}, which in
a \texttt{priority\_queue} yields \emph{smallest}-first; it was corrected to
\texttt{<} for the intended largest-first (longest-processing-time) ordering.
The decomposition is unchanged --- the queue reorders \emph{when} regions are
processed, not which exist --- so the leaf count stays exactly 5{,}608.

\paragraph{Measurement.} Four passes on both binaries: a wall-time A/B
(3~reps, alternating); a \texttt{quiet}\,=\,0 metrics run (leaves, per-thread
totals, outliers); a Nsight Systems capture (NVTX ranges, CUDA API, CUPTI
kernel); and a \texttt{quiet}\,=\,0 distribution run. Baseline \texttt{binary-v3-9point}
built in a worktree.

\section{Analysis}

\subsection{The wall does not move, and the GPU is not starved}

The A/B delta is $-0.55\%$ with overlapping ranges (Table~\ref{tab:pq}) --- a
wash. Table~\ref{tab:pq-nsys} rules out the two things one might fear: the
GPU-thread busy time is $\approx$46.5--46.9~s in \emph{both} (about 91\% of the
$\sim$51~s wall), so largest-first neither idles the GPU nor overloads it, and
total compute is conserved ($\approx$610~s, the same pixels). This is the
report-09 regime: a throughput-bound schedule whose load is already balanced to
under 2\,\% has no slack for a reordering to recover.

\subsection{The outlier is never routed to the GPU --- the premise fails}

The motivating hope was that largest-first would hand the $\sim$14~s interior
block to the GPU thread, where it would finish in milliseconds. It does not.
The CUPTI kernel \textbf{max is $\approx$139~ms in both} binaries
(Table~\ref{tab:pq-nsys}): the GPU's biggest region is the same $\sim$139~ms
region either way, so the $960\times540$ outlier was \emph{not} dispatched to
the GPU under the priority queue. The CPU worst-region actually rose slightly
(13.3~$\to$~14.4~s, Table~\ref{tab:pq}), and the CPU p99/max are unchanged
(Table~\ref{tab:pq-dist}) --- the big regions still run on CPU threads.

The mechanism is the \emph{shared} queue. All twelve workers pull from one
priority queue; when the outlier is enqueued, the next thread to call
\texttt{extract()} takes it, and with eleven CPU threads against one GPU thread
that is a CPU thread roughly 11:1. Largest-first changes the \emph{order} regions
are handed out, not \emph{which processor} gets them. Routing big work to the GPU
specifically would need GPU \emph{affinity} (a separate GPU lane, or the GPU
thread preferentially taking the largest while CPU threads take the smallest) ---
not a single shared largest-first queue.

\subsection{What largest-first does change (but it does not matter here)}

It does reshape the distribution as intended (Table~\ref{tab:pq-dist}): the GPU's
regions get slightly larger (median $3.64\to3.97$~ms; avg $11.0\to11.6$~ms over
fewer launches, $4247\to4004$) and the CPU gets more, smaller regions (count
$1515\to1659$, median $46.4\to33.0$~ms). This is the longest-processing-time
effect --- big items first, small items filling the tail. It would help makespan
on an \emph{imbalanced} schedule; here the FIFO schedule was already balanced to
under 2\,\%, so the reshaping is invisible at the wall.

\subsection{Would the outlier even belong on the GPU? Yes --- it is big \emph{and} coherent}

A natural objection to GPU affinity: these slow regions sit on the fractal
boundary, so wouldn't they cause heavy warp divergence and run poorly on the GPU
anyway? For \emph{these} regions the answer is no, and the reason is the same
property that makes them slow on the CPU.

\paragraph{They are big because they are interior, not because they are
boundary.} The outliers are $960\times540$ (depth~1, a quarter of the frame) and
$480\times270$ (depth~2); they stay large precisely because their corners are all
interior and uniform, so the heuristic never splits them. The genuinely
\emph{divergent} regions --- those straddling the boundary --- have large corner
spread and are therefore split all the way to the $240\times135$ pixel floor.
Big-and-coherent and small-and-divergent; the pathological big-and-divergent case
does not occur.

\paragraph{Warp divergence is low for an interior region.} A warp's runtime is
set by its slowest (highest-iteration) thread; threads that escape early simply
idle their lanes without adding work. Frame~73's region is 97.3\,\% interior, so
in a 32-thread warp $\sim$31 threads run to \texttt{maxIter} together --- only
$\sim$3\,\% of lanes are wasted. The cost is the \texttt{maxIter} arithmetic
itself (which the GPU does in parallel), not divergence. ``97.3\,\% interior''
here means 97.3\,\% of that region's pixels reach \texttt{maxIter} (in the set ---
specifically inside a \emph{minibrot}, since \texttt{cardioidBulb}\,=\,0, not the
main cardioid/bulb).

\paragraph{Direct evidence: same-size region, both processors.}

The GPU never received a $960\times540$ region (the shared queue sent them to CPU
$\sim$11:1), but it did process 183 interior $480\times270$ regions, the largest
at 139~ms --- versus $\sim$4~s for the same size on a CPU thread ($\sim$29$\times$).
The outlier, at $4\times$ the pixels, would be $\sim$0.45~s on the GPU versus
13.3~s on a CPU thread. So routing it to the GPU is a $\sim$30$\times$ win, not a
divergence trap.

\paragraph{Caveat: the kernel is double precision.} \texttt{diverge()} iterates
in \texttt{double}, and this GPU (Turing TU116) executes FP64 at 1:32 of FP32 ---
a $\sim$160~GFLOPS ceiling versus its $\sim$5~TFLOPS FP32 peak. Even the coherent
$480\times270$ region reaches only $\sim$40\,\% of that FP64 ceiling. Recasting the
iteration in \texttt{float} would be far faster on the GPU, but FP32's $\sim$7
significant digits cannot resolve the deepest frames (coordinates to $\sim$15
digits), so it would corrupt the late zoom; double is required for correctness,
and the GPU is still $\sim$30$\times$ faster than the serial CPU on these regions.
This reinforces recommendation~2: the outlier is the ideal GPU candidate ---
large, coherent, $\sim$30$\times$ faster --- and the shared priority queue's only
failure was the 11:1 extraction that kept it on the CPU.

\section{Methodology Note: Power-State Contamination and the AC Gate}

This experiment was nearly reported with the opposite conclusion. An initial run
showed the priority queue at $\sim$104~s and a later full pass put \emph{both}
binaries at $\sim$147~s --- an apparent 2$\times$ regression. The cause was not
the code: the laptop was on \textbf{battery}, and the GTX~1660~Ti~Max-Q had its
power limit clamped to \textbf{30~W} (default 60~W, \texttt{SW Power Cap}
active), so the GPU drew 14~W at 1140~MHz, processed far fewer regions
(GPU leaves $\approx$1{,}670 vs.\ the healthy $\approx$4{,}190), and the CPU pool
absorbed the rest. The CPU was also on a power-saving profile.

Two lessons. First, \textbf{never compare a fresh run against historical
numbers without confirming both ran in the same power state} --- the false
2$\times$ ``regression'' came from comparing a battery run against report-09's
AC numbers. Second, an \textbf{AC-power sanity gate} now precedes any timing
pass: one FIFO base run must return $\approx$52~s with GPU\,$\ge$\,4{,}000 leaves,
or the pass is aborted. After a reboot restored the 60~W limit, the gate passed
(50.5~s, GPU 4{,}167, GPU drawing 35.8~W at 1845~MHz mid-run) and the numbers
above were collected. All measurements in this report are post-gate.

\section{Recommendations}

\begin{enumerate}
    \item \textbf{Do not adopt the shared largest-first queue as the default.}
    It is wall-neutral ($-0.55\%$, within noise), decomposition-identical, and
    adds \texttt{priority\_queue} bookkeeping for no measured benefit. Keep the
    FIFO \texttt{deque} for simplicity. (If kept, it is harmless --- correctness
    is unchanged.)

    \item \textbf{If the goal is to offload the outlier to the GPU, use GPU
    affinity, not a shared queue.} Have the GPU thread preferentially pull the
    largest region (or a dedicated large-region lane) while CPU threads pull
    smaller ones. The $960\times540$ block costs $\sim$14~s on a CPU thread but
    only $\sim$139~ms on the GPU; routing \emph{it} specifically is the only
    version of this idea with real upside --- it would cut the worst-region tail
    by two orders of magnitude. A shared largest-first queue cannot do this
    (11:1 CPU:GPU extraction).

    \item \textbf{The binding lever remains work-reduction (report~09).} Even a
    perfect schedule cannot beat the conserved $\sim$610~s of compute; periodicity
    checking in \texttt{diverge()} or an interior certificate is what shrinks it.
\end{enumerate}

\section{Conclusion}

Replacing the FIFO work queue with a largest-first priority queue is
wall-neutral on this hybrid ($-0.55\%$, within noise). The Nsight Systems
breakdown shows the GPU is equally utilised either way and, decisively, that the
kernel max is unchanged at $\approx$139~ms --- the $\sim$14~s outlier is never
routed to the GPU, because a single shared queue hands big regions to whichever
of the twelve workers extracts next, a CPU thread $\sim$11:1. With the load
already balanced under 2\,\% and total compute conserved, reordering has nothing
to recover. The lever that would help --- routing the outlier to the
$\sim$40$\times$-faster GPU --- requires GPU \emph{affinity}, not a shared
priority queue; and the lever that bounds the wall remains work-reduction. The
experiment also reaffirmed a measurement discipline: a battery-induced 30~W GPU
cap had falsely shown a 2$\times$ regression, which an AC-power gate caught
before it reached a conclusion.
