\begin{abstract}
We re-measure the integrated-GPU backend of \texttt{20-igpu-opencl.tex} on a
binary that also carries the periodicity check of
\texttt{21-periodicity-check.tex}. Method: the same production A/B
(100 frames, $1920\times1080$, \texttt{diffT}\,=\,0.1, save=1, three
alternated reps), \texttt{dGPU+iGPU+10CPU} vs \texttt{dGPU+11CPU}.
\textbf{The iGPU's $-23.2\%$ collapses to a wash}: $28.05\pm0.12$\,s vs
$27.36\pm1.60$\,s ($+2.5\%$ on all-reps means, $-0.7\%$ excluding a cold
first run). The mechanism is an exact cancellation: the iGPU strips
27\,CPU-seconds off the binding CPU pool, and the CPU worker it displaces
would have contributed 26.5. Expressed as marginal value, periodicity
re-priced the dGPU from $\sim$23 to 2.7 CPU-worker equivalents and the iGPU
from $\sim$13 to $\approx$1. Two follow-up probes kill the obvious rescue:
13-thread oversubscription loses $+7.4\%$ (the lanes busy-spin), and with
\texttt{cudaDeviceScheduleBlockingSync} the recovered worker lands past the
SMT knee --- a wash that also closes the report-15 confirmation item.
Recommendation: keep \texttt{feat/igpu-opencl} off \texttt{main}.
\end{abstract}

\section{Background and question}

Report \texttt{20-igpu-opencl.tex} added a third executor --- an OpenCL FP64
backend on the Intel UHD 630 --- and measured a surprise: replacing one CPU
worker with the iGPU lane cut the production wall from 57.72 to 44.32\,s
($-23.2\%$). The mechanism was a second \emph{serial lane} for the big
coherent interior regions, stripping 187\,s of MAXITER-grind off the binding
CPU pool.

Report \texttt{21-periodicity-check.tex} then attacked that same grind
directly: an exact Brent cycle detector in the CPU \texttt{diverge()} lets an
interior pixel return after $\sim$900 iterations-equivalent instead of
10{,}000, byte-identically. The hybrid wall fell $57.6\to30.9$\,s and its
closing section flagged the consequence measured here: \emph{the iGPU's
$-23.2\%$ was priced against work that no longer exists}. This experiment
answers the re-opened question --- and, having both changes in one binary for
the first time, prices every device in the system against the new, faster
CPU worker.

\section{Methodology}

\paragraph{Binary.} \texttt{main} (carrying \texttt{binary-v6-periodicity})
merged into \texttt{feat/igpu-opencl} at commit \texttt{833a791}; the two
changes touch disjoint functions (\texttt{diverge()} vs the
\texttt{ExecKind} dispatch in \texttt{compute()}) and auto-merged. One
binary, both changes; all runs in this report use it. Probe 2 rebuilds with
one added line (\S\ref{sec:probe2}).

\paragraph{Workload and configurations.} Production \texttt{spec.in}
(100 frames, $1920\times1080$, deep zoom to \texttt{maxIter} 10{,}000),
\texttt{diffT}\,=\,0.1, \texttt{pixT}\,=\,32{,}768, quiet=1, save=1 --- the
same Fig.-11.15 methodology as the experiment-20 and 21 headlines. Headline
arms: mode 1 (\texttt{dGPU+11CPU}) vs mode 3 (\texttt{dGPU+iGPU+10CPU}), 12
threads total, three reps with the two modes alternated inside each rep so
thermal drift hits both equally. Hardware: i7-9750H (6C/12T) + GTX 1660 Ti
Max-Q + UHD 630 (\texttt{intel-compute-runtime-legacy1}), on AC power.

\paragraph{Calibration honesty.} Mode 1 here runs $\sim$9\% faster than
experiment 21's identical configuration measured one day earlier
($27.4$--$28.3$\,s steady-state vs $30.94\pm0.56$). The difference is thermal
history: experiment 21 interleaved 163-second CPU12 baseline grinds before
every hybrid run; this sweep is six $\sim$28\,s runs. Within-batch, the first
(coldest) run of the day is reproducibly the fastest by $1.5$--$3$\,s.
Absolute walls on this laptop therefore carry a $\sim$10\% thermal-history
band; every comparison in this report is same-batch and alternated, which the
band does not affect.

\section{Analysis}

\subsection{The swap arithmetic: relief equals forfeit}

Table~\ref{tab:split} is the whole result. The wall in both modes is set by
the CPU pool (busiest thread $\approx$ wall in all six runs). Swapping the
11th worker for the iGPU lane removes $292-266=27$ CPU-seconds from the
pool; the worker it displaced would have supplied $292/11=26.5$
CPU-seconds of capacity in the same wall time. The two sides of the trade
agree to within 2\%, which is exactly what a wash looks like from inside.

The reason is \emph{what} the iGPU takes. GPU affinity (report
\texttt{12-gpu-affinity.tex}) hands both GPU lanes the largest queued
regions, which at \texttt{diffT}\,=\,0.1 are certified-interior content ---
precisely the region class where the periodicity check makes a CPU worker
$\sim$11$\times$ faster (the pure-interior single-region test of experiment
21: $67.27\to5.98$\,s). The pool work that remains --- the truly-escaping
boundary band plus detection latency --- lives in small pixel-floor regions
the GPUs never pull. The iGPU is relieving the pool of the one thing the
pool is no longer slow at.

Contrast experiment 20: the same swap then stripped 187\,s off a 620\,s
pool, because interior grind at 10{,}000 iterations per pixel was the CPU's
pathology. That work class is gone; the win went with it.

\subsection{Periodicity re-priced every accelerator, not just the iGPU}
\label{sec:reprice}

The marginal value of a device can be expressed in CPU-worker equivalents
$k$ from the config walls, using the makespan model
$\text{wall} \propto W/(\text{effective workers})$ --- valid here because the
pool binds and the decomposition (hence $W$) is identical across configs.
Replacing one of 12 CPU workers with a lane of value $k$ gives
$k = 12\,({\text{wall}_{\text{CPU12}}}/{\text{wall}_{\text{lane}}}) - 11$.

The collapse factors ($8.7\times$ and $13\times$) bracket the pure-interior
elimination ratio ($11.2\times$), and that is not a coincidence: the GPUs'
diet is interior-dominated by construction (affinity feeds them the
certified-uniform largest regions), so their worker-equivalence scales down
by almost exactly the discount the CPU received on that content. The
important framing: \textbf{the iGPU did not get slower}. Its raw iteration
throughput is unchanged --- of order 15--20 scalar CPU threads. Its
\emph{marginal} value collapsed because report 21 removed 10/11 of the
iterations a CPU spends on the content the iGPU serves, while both GPU
kernels still grind all 10{,}000 (deliberately: report
\texttt{16-ncu-divergence.tex} showed the interior kernels are 100\%
warp-coherent and FP64-bound, so an early-out would introduce divergence).
The dGPU survives at $k=2.7$ because it is $\sim$1.7$\times$ faster per
region than the iGPU, runs its lane nearer saturation, and does not share
the CPU package's power budget.

\subsection{Probe 1: keeping all eleven workers by oversubscribing loses}

If the wash is just ``the lane costs a worker,'' the obvious rescue is to
pay with a thread the hardware does not have: 13 threads = 11 CPU workers
+ both lanes on 12 logical CPUs.

The reason is that the premise ``the GPU thread is idle while the kernel
runs'' is false on this system. Report
\texttt{15-async-streams-analysis.tex} established that the CUDA
synchronous wait \emph{busy-spins} (driver \texttt{ScheduleAuto} spins when
contexts $<$ cores), and both lanes additionally execute the host-side
commit (the per-pixel \texttt{setPixel} loop) --- so each lane genuinely
occupies a logical CPU. Thirteen runnable threads on twelve logical CPUs
just timeshare: the pool's busiest thread rises $26.5$--$27.9\to
28.2$--$29.0$\,s and the pool's busy-time counters inflate $265\to305$\,s
(the per-thread timers are wall-clock, so descheduled time counts). The 11th
worker's nominal 27\,s of capacity is consumed by the contention it creates.

\subsection{Probe 2: BlockingSync --- the recovered worker is past the SMT knee}
\label{sec:probe2}

Probe 1's failure is the spin's fault, so probe 2 removes the spin --- the
one-line confirmation report 15 proposed:

\begin{sourcecode}{cpp}{kernel.cu, top of \texttt{CUDAmemSetup()} (commit \texttt{29c68ea}) -- the driver now sleeps the dGPU lane on synchronous waits.}
// Make the driver sleep the calling thread on synchronous waits instead of
// busy-spinning (default ScheduleAuto spins when contexts < cores). Must be
// set before the first context-creating runtime call below.
CUDA_CHECK_RETURN(cudaSetDeviceFlags(cudaDeviceScheduleBlockingSync));
\end{sourcecode}

Three readings, in order of confidence:

\paragraph{BlockingSync is safe and roughly neutral.} Every configuration
shifted $\approx-1.6$\,s against the pre-BlockingSync batch, but the
first-of-batch cold effect is the same magnitude (24.33\,s here, 25.54\,s in
the morning batch), so we do not claim the shift. What the data does
establish: no regression --- sleep/wake latency is invisible at
20--70\,ms/region. This closes report 15's ``cheap confirmation'' item; the
line stays.

\paragraph{The recovered worker adds only past-the-knee capacity.} With the
lane sleeping, the 12th worker genuinely ran (pool busy 296\,s vs 263--276\,s
absorbed) --- but per-region CPU cost rose $53\to57$\,ms and the wall did not
move ($26.41$ vs $25.69$, overlapping spreads). The 12th worker lands on the
SMT sibling of an already-busy core, worth the $\sim$0.3--0.4 physical cores
report 15 priced for exactly this position --- while the busy-spinner it
displaced was already SMT-polite (a \texttt{PAUSE} loop yields core
resources to its sibling). The naive model ``freeing a spinning logical CPU
recovers $1/11$ of the pool'' double-counts what the spin never really took.

\paragraph{Mode 3 loses under every scheme tried.} Worker swap ($+2.5\%$),
spin oversubscription ($+7.4\%$), blocking oversubscription ($+8.0\%$ vs its
own control). The CUDA flag does not touch the OpenCL lane's wait, both
lanes still carry their host-side commits, and the iGPU shares the 45\,W
package budget with the pool it is supposed to be helping. This also closes
the ``one thread drives both GPUs'' multiplexer idea without writing it: its
entire prize is the recovered worker just measured at noise level.

\subsection{Limits}

The headline A/B is 3 reps; the probes are 2. Within-config scatter reaches
2.7\,s (cold-start), so the probes can rule out the $-8\%$-class effects they
hunted but cannot rank the $\sim$1\,s differences among surviving configs.
The $k$ model of \S\ref{sec:reprice} assumes a binding, work-conserving pool;
it is a characterization tool, and its post-v6 iGPU value is bounded by the
same noise as the wash it is derived from ($k_i\in[0.7,1.1]$). Mode 2
(\texttt{iGPU+11CPU}) was not re-run post-v6, and the OpenCL lane's
wait-vs-spin behaviour on the NEO runtime was not investigated --- neither
affects the merge decision, since both could only matter in configurations
that already lose.

\section{Recommendations}

\begin{enumerate}
  \item \textbf{Do not merge \texttt{feat/igpu-opencl} into \texttt{main}.}
        Measured value atop v6: $+2.5\%$/$-0.7\%$ (a wash) for the cost of an
        OpenCL ICD loader dependency at build time, the AUR legacy Intel
        runtime at run time, and a third executor in the code. The branch
        stays as a characterization --- report 20 documents the pre-v6 win,
        this report its repeal --- mirroring the reports-11/18 precedent.
  \item \textbf{Keep the \texttt{cudaDeviceScheduleBlockingSync} line} (it
        stays on this branch; porting to \texttt{main} is optional and
        expected to be worth $\approx0$ at 12 threads). It is measured safe,
        and it retires report 15's open confirmation.
  \item \textbf{The one lever left on this axis is GPU-side periodicity}
        (report 21, recommendation 3). \S\ref{sec:reprice} sharpens its
        motivation: the GPUs' value collapsed because only the CPU got the
        $\sim$11$\times$ interior discount. A GPU-side check would re-inflate
        $k$ for both GPUs --- against report 16's warning that it breaks the
        kernels' 100\% warp coherence. That trade must be measured, not
        assumed, and it is a separate A/B on \texttt{kernel.cu}.
  \item \textbf{If the iGPU is ever revisited} (e.g.\ after GPU-side
        periodicity changes the price again): the unmeasured residuals are
        the \texttt{scanLine} commit cheapening and strict
        dGPU-priority-on-largest (report 20 \S Analysis), plus mode 2 as a
        budget config for machines without a dGPU.
  \item \textbf{Treat absolute walls on this laptop as carrying a
        $\sim$10\% thermal-history band.} Same-day alternated controls (as
        used throughout here) are mandatory for any future headline number;
        cross-day comparisons of raw seconds are not meaningful at the
        current effect sizes.
\end{enumerate}

\section{Conclusion}

Experiment 22 answers the question report 21 re-opened: with the periodicity
check merged, the integrated GPU's $-23.2\%$ becomes $+2.5\%$/$-0.7\%$ --- a
wash, mechanism fully accounted (27 CPU-seconds relieved vs 26.5 forfeited,
per-worker load unchanged at 26.6\,s/thread). Two probes eliminated the
rescue paths: oversubscription loses to the busy-spinning lanes ($+7.4\%$),
and with BlockingSync the recovered worker is worth only past-the-SMT-knee
capacity (wash), which also retires the dual-GPU multiplexer idea and closes
report 15's confirmation item. The general lesson is the repricing table:
work-reduction on one executor devalues every other executor whose niche was
that work --- the dGPU fell from $\sim$23 to 2.7 worker-equivalents, the
iGPU from $\sim$13 to $\approx$1, both by roughly the interior
iteration-elimination ratio. \texttt{feat/igpu-opencl} should remain
unmerged; the next lever on this axis, if any, is giving the GPUs the same
discount the CPUs got.
