\begin{abstract}
This report documents the instrumentation added to \texttt{mandelHybrid},
a hybrid CPU+GPU Mandelbrot set renderer that uses adaptive region splitting
as its load-balancing strategy. Two complementary profiling mechanisms were
introduced: NVTX (NVIDIA Tools Extension) annotations, which label regions
of CPU code so that Nsight Systems can render them as colored spans on a
per-thread timeline, and CUDA event timing, which uses GPU-side hardware
timestamps to measure kernel execution and device-to-host transfer time
independently of any CPU clock. The results show that the GPU thread is
approximately 40$\times$ faster per region than a single CPU thread,
naturally pulling 77\% of all work-queue items. All 12 threads (1 GPU + 11
CPU) finish within roughly 2 seconds of each other, demonstrating that the
dynamic work queue achieves good load balance. The dominant finding is that
\texttt{cudaMemcpy} spends 99.9\% of its reported time waiting for the GPU
kernel to finish—the actual device-to-host data transfer takes only 15~µs on
average, not 13.4~ms as the CUDA API summary suggests.
\end{abstract}

% -----------------------------------------------------------------------
\section{Program Overview}

\texttt{mandelHybrid} renders a sequence of Mandelbrot frames using a
shared work queue. One \texttt{CalcThr} thread (thread~0) drives the GPU;
all remaining threads compute on the CPU. Every thread calls
\texttt{MandelRegion::examine()} in a loop, pulling items from the queue
until it is empty. \texttt{examine()} evaluates the four corners of a
region: if the corners are uniform (the spread of iteration counts
is small relative to the \texttt{diffThresh} parameter) or the region is
smaller than \texttt{pixelSizeThresh} pixels, it calls
\texttt{MandelRegion::compute()} immediately; otherwise it splits the
region into four equal quadrants and pushes them all back onto the queue.

The profiling run rendered 100 frames at 1920$\times$1080 pixels on a
NVIDIA GeForce GTX~1660~Ti Max-Q using 12 threads (1 GPU thread +
11 CPU threads), which is the machine's physical core count.

\section{Instrumentation Changes}

\subsection{Build System (Makefile)}

The only build-system change was adding \texttt{-ldl} to the linker flags:

\begin{sourcecode}{make}{Makefile — new linker flag.}
# NVTX3 (CUDA 11+) is header-only: it dlopen()s the profiler's runtime
# library when Nsight Systems is active, so no -lnvToolsExt is needed.
# -ldl provides the dlopen symbol that the NVTX3 headers use internally.
CC_LINK_FLAGS = -lm -lstdc++ $(QT_LIBS) -lpthread -ldl
\end{sourcecode}

\textbf{Why \texttt{-ldl}?} The NVTX3 API (CUDA~11+) is entirely
header-only. When the program starts under Nsight Systems, the profiler
injects a shared library (\texttt{libNvtxInjection64.so}) into the process
using \texttt{LD\_PRELOAD}. The NVTX3 headers then call \texttt{dlopen()}
at the first \texttt{nvtxRangePush} to locate the injected library and
register the annotation. Without linking \texttt{libdl}, the linker cannot
resolve \texttt{dlopen} and the build fails. When the program runs
without Nsight Systems, the \texttt{dlopen} call simply finds
nothing and all NVTX calls become no-ops with negligible overhead.

In CUDA~10 and earlier there was a real shared library
\texttt{libnvToolsExt.so} to link against. CUDA~11 replaced it with the
header-only NVTX3 design—the \texttt{nvtx3/nvToolsExt.h} header ships
inside the CUDA toolkit at \texttt{\$CUDAINST/include/nvtx3/}.

% ---
\subsection{NVTX Annotations}

\subsubsection{What NVTX Is}

NVTX (NVIDIA Tools Extension Library) is a C API for annotating host-side
code with ranges and markers. A range is an interval opened
with \texttt{nvtxRangePush("label")} and closed with \texttt{nvtxRangePop()}.
Ranges stack: if you push inside an existing range, the inner range appears
nested under the outer one. Nsight Systems records the wall-clock timestamps
of every push and pop, associates them with the calling thread, and renders
them as colored bars on a per-thread lane in the timeline view. This lets
you see—at a glance and without changing the program's logic—which thread is
doing what, for how long, and how CPU and GPU work interleave.

\subsubsection{Adding the Header}

The same header is included in both the CUDA translation unit and the plain
C++ translation unit:

\begin{sourcecode}{cuda}{kernel.cu — NVTX header.}
// nvToolsExt.h lives under nvtx3/ in CUDA 11+ installations.
#include <nvtx3/nvToolsExt.h>
\end{sourcecode}

\begin{sourcecode}{cpp}{mandelregion.cpp — NVTX header.}
// Including it in a plain C++ translation unit is fine;
// nvcc is not required to use the NVTX3 CPU-side API.
#include <nvtx3/nvToolsExt.h>
\end{sourcecode}

\subsubsection{Annotating \texttt{examine()}}

\texttt{examine()} is the innermost work-queue loop body executed by every
thread for every task. A range was added around the entire function:

\begin{sourcecode}{cpp}{mandelregion.cpp — examine() annotation.}
void MandelRegion::examine(WorkQueue &q, bool onGPU)
{
    // Wraps the full decision cycle: corner evaluation + compute or split.
    // In Nsight Systems this appears on the calling thread's row, letting
    // you count how many examine() calls each thread handles.
    nvtxRangePush("examine region");

    // ... corner evaluation and compute/split logic (unchanged) ...

    nvtxRangePop(); // closes "examine region"
}
\end{sourcecode}

Because \texttt{examine()} calls \texttt{compute()}, the CPU and GPU
compute ranges (described below) appear nested inside
\texttt{"examine region"} in the timeline. The gap between the outer and
inner range represents pure overhead: corner-iteration calls and queue
operations.

\subsubsection{Annotating \texttt{compute()} — CPU Path}

\begin{sourcecode}{cpp}{mandelregion.cpp — CPU compute annotation.}

{
    // ...
    else // CPU path
    {
        // Nsight Systems shows this as a colored bar on the calling

        // ... existing pixel loop ...

        nvtxRangePop(); // closes "CPU region compute"

\end{sourcecode}

\subsubsection{Annotating \texttt{hostFE()} — GPU Path}

\begin{sourcecode}{cuda}{kernel.cu — GPU compute annotation.}

\subsection{CUDA Event Timing}

\subsubsection{What \texttt{cudaEvent\_t} Is}

A \texttt{cudaEvent\_t} is an opaque handle to a hardware timestamp on the
GPU. The GPU has its own free-running clock, independent of any CPU clock.
When you call \texttt{cudaEventRecord(event)}, the CUDA runtime enqueues a
``stamp yourself'' command into a CUDA stream. The GPU executes this
command in stream order: it will not record the timestamp until all
previously enqueued work in that stream has completed. The CPU returns from
\texttt{cudaEventRecord} immediately—the stamping happens asynchronously.

After the work you want to measure is done, you call
\texttt{cudaEventSynchronize(event)}, which blocks the CPU until the GPU
has written that specific timestamp. Then \texttt{cudaEventElapsedTime(\&ms,
start, stop)} reads the delta in milliseconds from the GPU hardware. This
approach is more accurate than a CPU-side timer because:

\begin{itemize}
    \item The GPU runs asynchronously; a CPU timer started before a kernel
    launch would include driver dispatch latency, not just kernel execution
    time.
    \item \texttt{cudaMemcpy} with \texttt{cudaMemcpyDeviceToHost} is a
    blocking CPU call that first waits for any pending GPU work to finish
    and then performs the transfer. A CPU timer around it measures
    kernel wait time + transfer time, not transfer time alone.
    CUDA events placed around just the transfer report only the transfer.
\end{itemize}

\subsubsection{Declaring Four Events and Accumulators}

Four events are declared as file-scope statics in \texttt{kernel.cu} so
they persist across all calls to \texttt{hostFE()}:

\begin{sourcecode}{cuda}{kernel.cu — event handles and accumulators.}
static cudaEvent_t evKernelStart, evKernelStop;
static cudaEvent_t evMemcpyStart,  evMemcpyStop;

// Cumulative totals across all hostFE() calls on thread 0.
static double totalKernelMs = 0.0;
static double totalMemcpyMs = 0.0;
static long   totalRegions  = 0;
\end{sourcecode}

Using four events—two around the kernel, two around the memcpy—lets us
separate kernel execution time from transfer time. Without them we could
only measure the combined time reported by the blocking
\texttt{cudaMemcpy} call.

\subsubsection{Creating Events in \texttt{CUDAmemSetup()}}

\begin{sourcecode}{cuda}{kernel.cu — CUDAmemSetup(), event creation.}
extern "C" unsigned int *CUDAmemSetup(int maxResX, int maxResY)
{
    // ... existing cudaMallocPitch and cudaHostAlloc ...

    // Create CUDA events once; reuse them for every kernel call.
    // cudaEventCreate allocates GPU-side timer resources.
    CUDA_CHECK_RETURN(cudaEventCreate(&evKernelStart));
    CUDA_CHECK_RETURN(cudaEventCreate(&evKernelStop));
    CUDA_CHECK_RETURN(cudaEventCreate(&evMemcpyStart));
    CUDA_CHECK_RETURN(cudaEventCreate(&evMemcpyStop));

    return h_res;
}
\end{sourcecode}

\texttt{CUDAmemSetup()} is called once before any rendering begins, making
it the right place to allocate GPU-side timer resources. Creating them here
avoids paying the allocation cost (a GPU round-trip) on every one of the
3,543 kernel launches.

\subsubsection{Recording Timestamps in \texttt{hostFE()}}

\begin{sourcecode}{cuda}{kernel.cu — hostFE(), full event timing block.}
// Record a GPU timestamp immediately before the kernel launch.
// The stamp executes after all previously-queued work finishes.
CUDA_CHECK_RETURN(cudaEventRecord(evKernelStart));

mandelKernel<<<grid, block>>>(d_res, upperX, upperY,
                               stepX, stepY, resX, resY, ptc, MAXITER);

// Stamp the moment the kernel finishes (still async -- resolved below).
CUDA_CHECK_RETURN(cudaEventRecord(evKernelStop));

// Stamp before the device-to-host copy.
CUDA_CHECK_RETURN(cudaEventRecord(evMemcpyStart));

CUDA_CHECK_RETURN(
    cudaMemcpy(h_res, d_res, resY * ptc, cudaMemcpyDeviceToHost));

CUDA_CHECK_RETURN(cudaEventRecord(evMemcpyStop));

// Block the CPU until the GPU has written all four timestamps.
// After this call, cudaEventElapsedTime gives accurate durations.
CUDA_CHECK_RETURN(cudaEventSynchronize(evMemcpyStop));

float kernelMs = 0.0f, memcpyMs = 0.0f;
CUDA_CHECK_RETURN(
    cudaEventElapsedTime(&kernelMs, evKernelStart, evKernelStop));
CUDA_CHECK_RETURN(
    cudaEventElapsedTime(&memcpyMs, evMemcpyStart, evMemcpyStop));

totalKernelMs += kernelMs;
totalMemcpyMs += memcpyMs;
totalRegions++;

// Per-region line printed to stderr.
fprintf(stderr,
    "[GPU region %4ld] size %4dx%4d  kernel %.3f ms  D->H %.3f ms\n",
    totalRegions, resX, resY, kernelMs, memcpyMs);
\end{sourcecode}

The key subtlety is that \texttt{cudaEventRecord(evKernelStop)} is called
on the CPU immediately after the kernel launch returns, not after
the kernel finishes. Because the default CUDA stream serializes all
operations, the GPU will record \texttt{evKernelStop} only after
\texttt{mandelKernel} completes. Similarly, \texttt{evMemcpyStart} is
stamped after the kernel finishes but before the bytes start moving,
\texttt{evMemcpyStop} after the last byte arrives. The
\texttt{cudaEventSynchronize(evMemcpyStop)} call is the single point
where the CPU blocks, ensuring all four timestamps have been committed
before \texttt{cudaEventElapsedTime} reads them.

\subsubsection{Printing the Summary in \texttt{CUDAmemCleanup()}}

\begin{sourcecode}{cuda}{kernel.cu — CUDAmemCleanup(), summary and teardown.}
extern "C" void CUDAmemCleanup()
{
    // Accumulated GPU-side timing. These numbers exclude host-side overhead.
    fprintf(stderr,
        "[GPU profiling summary]\n"
        "  Regions computed on GPU : %ld\n"
        "  Total kernel time       : %.3f ms  (avg %.3f ms/region)\n"
        "  Total D->H memcpy time  : %.3f ms  (avg %.3f ms/region)\n",
        totalRegions,
        totalKernelMs, totalRegions ? totalKernelMs / totalRegions : 0.0,
        totalMemcpyMs, totalRegions ? totalMemcpyMs / totalRegions : 0.0);

    cudaEventDestroy(evKernelStart);
    cudaEventDestroy(evKernelStop);
    cudaEventDestroy(evMemcpyStart);
    cudaEventDestroy(evMemcpyStop);

    CUDA_CHECK_RETURN(cudaFreeHost(h_res));
    CUDA_CHECK_RETURN(cudaFree(d_res));
}
\end{sourcecode}

\subsection{CPU Thread Timing with \texttt{clock\_gettime}}

\subsubsection{Thread-Local Accumulators}

\begin{sourcecode}{cpp}{mandelregion.cpp — thread-local state.}
#include <time.h>

// Each thread gets its own independent copy, initialized to zero.
// No mutex is needed because no two threads share the same copy.
static thread_local long   cpuRegionCount = 0;
static thread_local double cpuTotalMs     = 0.0;

static inline double elapsedMs(const struct timespec &t0,
                                const struct timespec &t1)
{
    return (t1.tv_sec  - t0.tv_sec)  * 1e3
         + (t1.tv_nsec - t0.tv_nsec) * 1e-6;
}
\end{sourcecode}

The \texttt{thread\_local} storage class ensures each \texttt{CalcThr}
thread has a private \texttt{(cpuRegionCount, cpuTotalMs)} pair. With 11
CPU threads running concurrently, a shared counter would require a mutex;
\texttt{thread\_local} eliminates all synchronization from the hot path.

\texttt{CLOCK\_MONOTONIC} is used instead of \texttt{CLOCK\_REALTIME}
because the monotonic clock never jumps backwards due to NTP corrections
or daylight saving changes.

\subsubsection{Bracketing the Pixel Loop in \texttt{compute()}}

\begin{sourcecode}{cpp}{mandelregion.cpp — CPU timing in compute().}
else // CPU path
{
    nvtxRangePush("CPU region compute");

    struct timespec t0, t1;
    clock_gettime(CLOCK_MONOTONIC, &t0);

    for (int i = 0; i < pixelsX; i++)
        for (int j = 0; j < pixelsY; j++)
        {
            // ... diverge() and setPixel() (unchanged) ...
        }

    clock_gettime(CLOCK_MONOTONIC, &t1);
    double ms = elapsedMs(t0, t1);
    cpuTotalMs += ms;
    cpuRegionCount++;

    fprintf(stderr,
        "[CPU region %4ld] size %4dx%4d  compute %.3f ms\n",
        cpuRegionCount, pixelsX, pixelsY, ms);

}
\end{sourcecode}

\subsubsection{Per-Thread Summary Function}

A new static method was declared in \texttt{mandelregion.h} and
implemented in \texttt{mandelregion.cpp}:

\begin{sourcecode}{cpp}{mandelregion.cpp — printCPUSummary().}
void MandelRegion::printCPUSummary()
{
    // GPU thread: cpuRegionCount == 0, nothing to print.
    if (cpuRegionCount == 0) return;
    fprintf(stderr,
        "[CPU profiling summary - this thread]\n"
        "  Regions computed on CPU : %ld\n"
        "  Total compute time      : %.3f ms  (avg %.3f ms/region)\n",
        cpuRegionCount,
        cpuTotalMs, cpuTotalMs / cpuRegionCount);
}
\end{sourcecode}

This function is called from \texttt{CalcThr::run()} once the work queue
drains, meaning it executes exactly once per thread at the natural end of
that thread's work:

\begin{sourcecode}{cpp}{main.cpp — calling printCPUSummary at thread exit.}
void CalcThr::run()
{
    MandelRegion *t;
    while ((t = que->extract()) != NULL)
    {
        t->examine(*que, isGPU);
        delete t;
    }
    // Work queue empty: print this thread's accumulated CPU timing.
    MandelRegion::printCPUSummary();
}
\end{sourcecode}

\section{Analysis}

\subsection{GPU is 40$\times$ Faster Per Region}

Comparing the average per-region compute times:

\begin{itemize}
    \item GPU: 13.5~ms average (from NVTX \texttt{"GPU region compute"})
    \item CPU: 535~ms average (from NVTX \texttt{"CPU region compute"}, summed across all 11 CPU threads)
\end{itemize}

The GPU is roughly 40$\times$ faster per region. Because all
threads compete for the same work queue, the GPU thread cycles back to
pull a new region 40$\times$ more often, naturally absorbing 77\% of all
leaf regions (3{,}543 out of 4{,}603).

This ratio is not a tunable parameter—it is determined by the compute
capability of the GPU versus the collective throughput of the CPU cores for
the Mandelbrot iteration kernel.

\subsection{Load Balance Is Good, But Not Perfect}

The per-thread total compute times (Table~\ref{tab:cpu-threads}) span
50.50~s to 52.59~s across the 11 CPU threads—a spread of only 4\%.
The GPU thread finishes its work in 47.1~s, roughly 3–5~s earlier than the
last CPU thread. The dynamic work queue achieves its goal: no thread sits
idle for long stretches while others have work.

The remaining 4\% imbalance is not caused by a structural flaw in the
scheduler. It arises entirely from per-region variance: some
Mandelbrot regions near the set's boundary require far more iterations than
the corner heuristic predicted. A thread that happens to pull such a region
will take much longer on it than a thread pulling a uniform interior region,
even if both regions have the same pixel count.

\subsection{The D$\rightarrow$H Memcpy Is Not the Bottleneck}

This is the most surprising finding. The CUDA API summary reports
\texttt{cudaMemcpy} as consuming 99.3\% of all CUDA API time (13.4~ms average
per call). However, the CUPTI hardware counters tell a different story: the
actual device-to-host transfer takes only 15~µs on average and
53~ms total across all 3{,}543 copies. The discrepancy is explained
by the semantics of \texttt{cudaMemcpy}:

\begin{enumerate}
    \item \texttt{cudaMemcpy} with \texttt{cudaMemcpyDeviceToHost} is a
    blocking, synchronous call.
    \item It first waits for all previously-enqueued GPU work to finish
    (i.e., it implicitly synchronizes the default stream).
    \item Only then does it initiate the DMA transfer.
\end{enumerate}

So the 13.4~ms reported by the CUDA API is almost entirely
kernel execution time (confirmed by the GPU kernel summary:
13.3~ms average). The actual PCIe transfer consumes only 15~µs. At
599.9~MB across 3{,}543 copies the effective bandwidth is
$599.9 / 0.053 \approx 11.3$~GB/s, which is healthy use of a PCIe~3.0
$\times$16 link (theoretical maximum: 16~GB/s). Optimizing memory layouts
or using pinned memory more aggressively would have essentially no impact.

This finding also highlights a missed optimization: during those
13.4~ms where the CPU is blocked in \texttt{cudaMemcpy} (which is really
waiting for the kernel), thread~0 (the GPU thread) cannot pull new work
from the queue. Using asynchronous CUDA streams—launching the kernel into
a non-default stream, overlapping the previous copy with the next kernel
launch—would allow thread~0 to stay in the queue-pulling loop and increase
GPU utilization.

\subsection{High Per-Region Variance}

\begin{itemize}
    \item GPU regions: min~8.2~µs, avg~13.3~ms, max~465.3~ms
    (57{,}000$\times$ range)
    \item CPU regions: min~0.9~ms, avg~535~ms, max~15.3~s
    (17{,}000$\times$ range)
\end{itemize}

The 15.3~s CPU outlier is a single large, complex region where the four
corners happened to agree (low iteration-count spread), so the heuristic
classified it as uniform and did not split it. However, the interior
contained many boundary pixels that required near-\texttt{MAXITER}
iterations. This single region is responsible for almost the entire 4\%
thread imbalance. Lowering \texttt{diffThreshold} (currently 0.5) would
force more splits on ambiguous regions, reducing worst-case outliers at the
cost of a higher total number of work-queue items.

\subsection{Examine Overhead Is Negligible}

The total time attributed to \texttt{"examine region"} minus the time
attributed to the two compute ranges gives the pure overhead of the
adaptive subdivision logic (corner evaluation and queue operations):

\[
    639.1 - 567.2 - 47.8 \approx 24.1~\text{s across 6{,}104 calls}
    \approx 4~\text{ms per call}
\]

Four milliseconds per \texttt{examine()} call for up to four
\texttt{diverge()} corner evaluations is a small overhead relative to the
compute times involved.

\subsection{Summary of Findings and Recommended Improvements}

\begin{table}[H]
    \centering
    \caption{Key findings and their implications.}
    \begin{tabular}{p{5.5cm} p{7cm}}
        \hline
        Finding & Implication \\
        \hline
        GPU 40$\times$ faster per region &
            It naturally pulls 77\% of work-queue items \\
        All threads finish within $\sim$2~s of each other &
            Dynamic work queue achieves good load balance \\
        D$\rightarrow$H transfer: 15~µs avg, not 13.4~ms &
            PCIe bandwidth is not the bottleneck; kernel time dominates \\
        GPU thread blocks 13.4~ms per region in \texttt{cudaMemcpy} &
            Async streams could recover this idle time \\
        15.3~s CPU outlier region &
            Corner heuristic misclassified a complex region; lower \texttt{diffThreshold} \\
        \texttt{examine()} overhead: $\sim$4~ms/call &
            Subdivision logic is cheap; not worth optimizing \\
        \hline
    \end{tabular}
\end{table}

Three concrete improvements follow from the analysis:

\begin{enumerate}
    \item \textbf{Asynchronous CUDA streams.} Launch \texttt{mandelKernel}
    into a named stream and use \texttt{cudaStreamSynchronize} only when
    reading results. This lets thread~0 call \texttt{que->extract()} and
    begin preparing the next region while the previous kernel executes and
    copies, recovering the $\sim$13~ms per-region idle time.

    \item \textbf{Tighter \texttt{diffThreshold}.} The default value of 0.5
    is permissive. Reducing it to 0.2--0.3 would force more splits on
    ambiguous boundary regions, capping the worst-case per-region time and
    improving tail-latency balance across CPU threads.

    \item \textbf{Largest-first work queue.} The \texttt{MandelRegion::Compare}
    functor already exists in \texttt{mandelregion.h} for this purpose.
    Replacing \texttt{WorkQueue}'s \texttt{deque} with a \texttt{priority\_queue}
    (largest pixel area first) would give the GPU thread preferentially
    larger regions, which it handles fastest, and leave smaller regions for
    CPU threads, reducing the chance of a large outlier blocking a CPU
    thread near the end.
\end{enumerate}
