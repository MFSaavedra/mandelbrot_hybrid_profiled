\begin{abstract}
This report implements and measures the work-reduction lever that every report
since \texttt{09-9point-sampling.tex} has pointed at: \textbf{periodicity
checking} in the CPU \texttt{diverge()}. A Brent-style cycle detector saves the
orbit state at doubling intervals and returns \texttt{MAXITER} as soon as the
floating-point state \emph{exactly} revisits the saved state --- an
exactly-repeating orbit of a deterministic map can never escape, so the early
return is provably exact. Measured on the production spec (100 frames,
$1920\times1080$, \texttt{diffT}\,=\,0.1, \texttt{save}\,=\,1, 3 alternating
reps, on AC): \textbf{CPU12 falls $164.4 \to 35.2$~s ($-78.6\%$, $4.67\times$)
and the best hybrid falls $57.6 \to 30.9$~s ($-46.3\%$, $1.86\times$)} --- a
larger win than every scheduling lever combined (affinity $-3.6\%$, report~12;
second GPU $-23.2\%$, report~20), from 18 added lines in one function. The
mechanism is work \emph{elimination}: the CPU pool's compute drops
$1{,}917 \to 402$~s while the 5{,}608-leaf decomposition and all 100 output
frames stay \textbf{byte-identical}. The hybrid queue self-rebalances --- CPU
threads, now $\sim$5$\times$ faster on interior content, take 4{,}279 regions
(was 2{,}160) and the GPU keeps only the 1{,}329 largest. The CPU pool remains
the binding resource (busiest thread 29.5~s $\approx$ wall), but what is left
on it is detection latency plus the genuinely-escaping boundary band, which no
exact early-out can skip.
\end{abstract}

\section{Motivation: 99.5\% of the iterations prove what a cycle detector knows early}
\label{sec:motivation}

Report~09 established that splitting \emph{conserves} total compute
($\sim$495~G iterations regardless of decomposition) and report~12 left the
system with both executors near saturation --- scheduling was exhausted. What
no report had quantified is \emph{what the iterations are for}. Parsing the
per-region work metrics of the report-12 production run
(\texttt{experiments/12-gpu-affinity/logs/dist\_aff.stderr}: all 5{,}608
leaves with size, mean iterations and interior fraction) gives

Weighting each leaf's interior pixels by the frame's \texttt{MAXITER} shows
that \textbf{99.5\% of all 495~G iterations are spent on interior pixels} ---
pixels whose orbits never escape, grinding to the iteration cap only to
confirm it. The entire boundary band plus all exterior costs 0.5\%. The wall
itself is the CPU pool grinding that content: in the report-12 run the 11 CPU
threads spend 545~s of compute, \textbf{99.0\% of it on interior pixels},
at 252~M iterations/s/thread --- content the GPU runs at 7.9~G iterations/s
($31\times$) but cannot absorb (kernel already 91\% busy, report~12). Interior
pixels are exactly the ones whose orbits settle into an attracting cycle, so a
cycle detector can return the same answer after the transient dies out instead
of after \texttt{MAXITER} iterations. That is the lever implemented here.

\section{The change: an exact Brent cycle detector}
\label{sec:change}

\texttt{MandelRegion::diverge()} gains a saved orbit state $(s_x, s_y)$,
refreshed at doubling intervals (iterations 8, 16, 32, \dots); every iteration
compares the new state against it with \emph{exact} floating-point equality:

\begin{sourcecode}{cpp}{mandelregion.cpp --- the modified \texttt{diverge()} inner loop (commit \texttt{fa6d8ac}).}
double sx = vx, sy = vy;
int nextSave = 8;
while (iter < MAXITER && (vx * vx + vy * vy) < 4)
  {
    tx = vx * vx - vy * vy + cx;
    ty = 2 * vx * vy + cy;
    vx = tx;
    vy = ty;
    iter++;
    if (vx == sx && vy == sy)
      return MAXITER;           // exact cycle: the orbit can never escape
    if (iter == nextSave)
      {
        sx = vx;                // Brent: refresh the anchor at iterations
        sy = vy;                // 8, 16, 32, ... so a period-p cycle entered
        nextSave *= 2;          // after a transient t is caught in O(max(p,t))
      }
  }
return iter;
\end{sourcecode}

\paragraph{Why this is exact.} The iteration is a deterministic map on the
floating-point numbers: the successor of $(v_x, v_y)$ is a pure function of
$(v_x, v_y, c_x, c_y)$. If the state ever equals a previously visited state,
the orbit is in a cycle \emph{of the FP map} and will repeat forever --- the
escape test can never fire, so the unmodified loop would have ground to
\texttt{MAXITER} and returned \texttt{MAXITER}. Returning it at detection time
changes \emph{when} the answer arrives, never \emph{what} it is. Conversely,
an orbit that eventually escapes can never exactly revisit a state (it would
otherwise cycle and not escape), so no escaping pixel can be misclassified:
there are no false positives \emph{by construction}, not by tolerance tuning.
This is why exact equality is used instead of an epsilon --- an epsilon
comparison would detect sooner but could constant-fill slowly-diverging
boundary orbits, changing the image. Section~\ref{sec:verify} confirms the
guarantee empirically: all 100 frames byte-identical, decomposition unchanged.

\paragraph{Detection latency.} Brent's doubling schedule catches a cycle of
period $p$ entered after a transient of $t$ iterations within $O(\max(p, t))$:
once the anchor lands inside the cycle and the window $\ge p$, the comparison
must hit. Figure~\ref{fig:orbit} traces the detector on an interior point of
the period-3 bulb: the FP orbit converges geometrically to the attracting
3-cycle, the distance-to-anchor sawtooth collapses by $\sim$3 decades per
checkpoint window, and at iteration 67 the state is \emph{bitwise} equal to
the anchor --- 67 iterations instead of the frame's thousands.

             skipping every remaining iteration. \label{fig:orbit}}

\paragraph{Cost.} The check adds two compares and a predictable branch to a
loop of $\sim$10 FP operations, paid by \emph{every} iteration, including
escaping pixels that never cycle. This bounds the worst case: exterior and
boundary-band pixels carry 0.5\% of all iterations
(\S\ref{sec:motivation}), so even a 30\% per-iteration slowdown on that slice
is $<$0.2\% of wall --- invisible next to the interior savings.

\paragraph{CPU branch only.} Report~16 measured the GPU's interior kernels at
exactly 100.0\% warp-execution efficiency and 85.6\% of FP64 peak: pixels in
a warp bail out at \emph{different} iterations under periodicity checking, so
the same change on the GPU would introduce divergence into a
perfectly-coherent FP64-bound kernel. \texttt{kernel.cu} is untouched; a GPU
variant is a separate A/B (recommendation~3).

\section{Methodology}
\label{sec:method}

\begin{itemize}
  \item \textbf{Hardware/software}: i7-9750H (6C/12T) + GTX~1660~Ti~Max-Q,
        same laptop as all prior reports; on AC power, rebooted before the
        session.
  \item \textbf{Binaries}: baseline built from \texttt{main} (\texttt{3061307})
        in a clean git worktree; treatment from \texttt{fa6d8ac} (=
        \texttt{main} + the \texttt{diverge()} change only --- an 18-line diff
        in one function).
  \item \textbf{Workload}: production \texttt{spec.in} --- 100 frames,
        $1920\times1080$, deep zoom to \texttt{maxIter} 10{,}000;
        \texttt{diffT}\,=\,0.1, \texttt{pixT}\,=\,32{,}768,
        \texttt{quiet}\,=\,1, \texttt{save}\,=\,1 (the Fig.~11.15 methodology,
        same as the experiment-20 headline).
  \item \textbf{Configs}: CPU12 (mode~0; the change carries everything) and
        dGPU+11CPU (mode~1; the production config). 3 reps each,
        \textbf{binaries alternated inside every rep} so thermal drift hits
        both sides equally (\texttt{experiments/21-periodicity-check/ab.sh}).
  \item \textbf{Calibration}: the baseline reproduces its references ---
        CPU12 164.4~s vs.\ experiment~20's 161.6~s ($+1.7\%$, third rep
        warmest); hybrid 57.6~s vs.\ 57.7~s ($-0.1\%$). The A/B is internally
        consistent regardless.
\end{itemize}

\paragraph{Incidental finding: spec-prefix buffer overflow.} Smoke-testing
with a spec whose image prefix was an absolute scratch path produced garbage
timings in \emph{both} binaries: \texttt{main.cpp} reads the prefix with
\texttt{fin >{}> imageFilePrefix} into \texttt{char[MAXFNAME-8]} (42 bytes),
so any prefix over 41 characters silently smashes the stack --- observed as
\texttt{maxIterations[]} corrupted to 24 in one binary and to a huge value in
the other (layout-dependent). Production specs use the 3-character prefix
\texttt{img}, so \textbf{no historical measurement is affected}. All runs in
this report use short prefixes; a bounds-checked read should land as its own
change on \texttt{main}.

\section{Analysis}
\label{sec:analysis}

\subsection{The win is work elimination, and it lands exactly where \S\ref{sec:motivation} said the work was}

CPU12 pool compute drops $1{,}917 \to 402$~s ($-79\%$), against the 99.5\%
interior-work share measured in \S\ref{sec:motivation}. The reason the two
numbers agree is direct: every interior pixel used to cost the frame's
\texttt{MAXITER} (up to 9{,}901) iterations; with the detector it costs the
transient-plus-period of its orbit. The residual 402~s is the boundary band
(escaping pixels cannot be skipped --- they never cycle) plus detection
latency. The implication closes the arc of reports 09--15: after scheduling
was exhausted (affinity $-3.6\%$, shared queue $\pm 0$, async streams priced
at $<$1\%), the first \emph{work} lever is worth $12\times$ more than the best
scheduling lever.

\subsection{Detection latency on the production content is $\sim$900 iterations-equivalent, not 67}

The single-region micro-benchmark (the zoom target rendered as one
pure-interior $1920\times1080$ region on one thread) runs $67.27 \to 5.98$~s
($11.2\times$). At the measured 308~M iterations/s that is $\sim$890
iterations-equivalent per pixel --- far above the 67 of
Figure~\ref{fig:orbit}'s period-3 point. This is consistent with the content:
the production zoom terminates inside a \emph{deep minibrot} (report~08),
whose attracting cycles have high periods and long transients, and Brent's
schedule pays $O(\max(p,t))$ with up to a $2\times$ overshoot from the
doubling window. The implication: the speed-up is content-dependent ---
shallow interior (main cardioid, low-period bulbs; the report-13
\texttt{inside} and \texttt{seahorse} regimes) will detect in tens of
iterations, while deep minibrot interior keeps hundreds --- but it is bounded
below by the exactness guarantee: the detector never does \emph{worse} than
the grind by more than the 2-compare overhead.

\subsection{The hybrid self-rebalances; the CPU pool remains the (much lower) wall}

No scheduling code changed, yet the GPU's share fell from 3{,}448 to 1{,}329
regions (Table~\ref{tab:split}). The reason is the min-max queue of report~12
reacting to the new cost structure: CPU threads drain the small end
$\sim$5$\times$ faster, reach deeper into the queue, and leave the GPU only
the regions where it still wins big (its average region grows $12.4 \to
19.6$~ms --- it keeps the largest). Busy times land near-matched: 25.7~s of
kernel vs.\ 29.5~s per CPU thread against a 30.9~s wall --- the CPU pool is
still the binding resource, but barely, and the balance re-found itself. The
implication for report~20: the iGPU's $-23.2\%$ was priced against a CPU pool
grinding interior at full cost; with that grind mostly gone, the marginal
value of a second GPU lane must be re-measured before merging
\texttt{feat/igpu-opencl} (recommendation~2).

\subsection{What the check cannot touch, and what that bounds}

Three costs remain by construction: (i) escaping pixels --- the boundary band
--- which never cycle and now dominate the CPU pool's residual 402~s
(CPU-only); (ii) detection latency, reducible only by a cheaper or
earlier-firing detector (an epsilon tolerance --- rejected here because it
forfeits the exactness guarantee); (iii) the per-iteration check overhead,
bounded at $<$0.2\% of wall by the 0.5\% work share of non-interior content.
Not measured: an exterior-dominated workload (the report-13 \texttt{outside}
regime) where the check only costs; the bound says the cost is the two-compare
overhead on cheap pixels, but the honest statement is that this report
measured the production zoom and its interior-heavy relatives only.

\section{Recommendations}
\label{sec:recs}

\begin{enumerate}
  \item \textbf{Merge \texttt{feat/periodicity-check} to \texttt{main} and tag
        \texttt{binary-v6-periodicity}.} It is the new best binary: hybrid
        $57.6 \to 30.9$~s ($-46.3\%$), CPU12 $164.4 \to 35.2$~s ($-78.6\%$),
        output byte-identical, 18 lines in one function. Trade-off: none
        measured --- the worst case is the $<$0.2\% overhead bound on
        exterior content.
  \item \textbf{Re-run the experiment-20 A/B on top of v6 before merging
        \texttt{feat/igpu-opencl}.} The iGPU's $-23.2\%$ was measured against
        a CPU pool that no longer exists ($620 \to 324$~s of grind). Expected
        outcome: the mode-3 win shrinks substantially --- the serial tail of
        big interior regions that motivated the second lane is now $5\times$
        cheaper to clear; whether it still pays for displacing a CPU worker
        is an open measurement. Cost: one $\sim$15-minute sweep.
  \item \textbf{A/B a GPU-side periodicity variant separately --- expect a
        loss.} Report~16 measured the interior kernels at 100.0\% warp
        efficiency and 85.6\% of FP64 peak; per-pixel early-out would
        introduce divergence into that kernel and add FP64 compare work to
        the bottleneck pipe. Measure once for the record; do not merge
        blind.
  \item \textbf{Fix the spec-prefix overflow on \texttt{main}.} Replace the
        unbounded \texttt{fin >{}> imageFilePrefix} with a bounded read
        (\texttt{std::string} + length check). Zero performance impact;
        removes a silent-corruption trap that cost this session its first
        smoke measurements.
  \item \textbf{Re-price the experiment-19 threshold sweep on v6.} The
        partial sweep's preliminary optimum (\texttt{pixT}\,=\,131{,}072)
        was driven by per-region overhead against interior grind; with
        interior $5\times$ cheaper on the CPU, the optimum may move again.
        Cost: the remaining 55 runs, now $\sim$3$\times$ faster on the
        treatment side.
\end{enumerate}

\section{Conclusion}
\label{sec:conclusion}

An 18-line exact cycle detector in the CPU \texttt{diverge()} --- anchor state
refreshed at doubling intervals, return \texttt{MAXITER} on exact FP revisit
--- eliminates most of the interior grind that \S\ref{sec:motivation} showed
to be 99.5\% of all iterations: \textbf{CPU12 $-78.6\%$ (164.4 to 35.2~s),
hybrid $-46.3\%$ (57.6 to 30.9~s, the new best wall)}, with all 100 frames
byte-identical and the 5{,}608-leaf decomposition unchanged --- the exactness
argument (an exactly-repeating orbit cannot escape; an escaping orbit cannot
exactly repeat) held empirically. The win is work elimination at fixed
balance: pool compute $-79\%$, the min-max queue re-balances itself around the
new costs, and the CPU pool remains the binding resource at a quarter of its
former cost. What remains is the truly-escaping boundary band, the
content-dependent detection latency ($\sim$900 iterations-equivalent inside
the deep minibrot vs.\ 67 in a period-3 bulb), and two follow-up measurements
this result re-opens: the iGPU's marginal value (report~20's baseline no
longer exists) and the threshold-sweep optimum.
