\begin{abstract}
Every prior report measured one zoom target --- the canonical minibrot at
$c\approx(0.0016,-0.8225)$. This report profiles the current best binary (GPU
affinity, report~\texttt{12-gpu-affinity.tex}) on \textbf{four qualitatively
different targets}: fully outside the set, fully inside the main cardioid, and
two boundary points distinct from the canonical one. The same wide first frame
and \texttt{maxIter} ramp are used; only the deep target differs. The headline
is a paradox: end-to-end cost spans \textbf{32$\times$} (2.37~s to 76.3~s) from
the same binary and parameters, yet the per-thread CPU spread stays under
\textbf{1.4\,\% in every regime} --- the dynamic queue is not the bottleneck
anywhere. The binding resource instead \emph{shifts}: light regimes are
trivially CPU-bound, heavy ones are GPU-saturated (\textbf{95--96\,\%}). The
most useful new finding is the seahorse point, which reproduces the interior
outlier \textbf{eightfold}, and all eight are \texttt{cardioidBulb}=1
main-cardioid interiors that overflow to the CPU because the GPU is saturated
--- exactly the case the cheap cardioid certificate would fix, unlike the
canonical minibrot (\texttt{cardioidBulb}=0). The outliers never bind the wall:
the schedule is throughput-bound.
\end{abstract}

\section{Setup and the Four Targets}

\paragraph{Workload and machine.} As in reports~09, 11 and~12: \texttt{spec.in}
of 100~frames at $1920\times1080$, \texttt{maxIterations} ramping $100\to10{,}000$,
on an i7-9750H (6c/12t) + GTX~1660~Ti~Max-Q, 12~threads (1~GPU~+~11~CPU),
\texttt{diffThreshold}\,=\,0.1, \texttt{pixelSizeThresh}\,=\,32{,}768,
\texttt{save}\,=\,0, on AC power. One rep per point (the per-thread spreads below
show the within-run balance directly).

\paragraph{The experimental variable.} Every run keeps the identical wide
full-set first frame ($-1.9,0.94$)--($1.2,-0.94$) at \texttt{maxIter}\,=\,100 and
changes \emph{only} the deep target. This makes the zoom target the single
independent variable and keeps every run comparable to the others and to the
canonical data of prior reports.

\begin{sourcecode}{bash}{The four targets -- the deep-zoom centre c (spec.in line 3).}
outside     : c = ( 1.0,        1.0       )   -- exterior, escapes at iter 2
misiurewicz : c = (-0.77568377, 0.13646737)   -- boundary, thin self-similar spiral
seahorse    : c = (-0.745428,   0.113009  )   -- boundary, main-cardioid crossing
inside      : c = (-0.5,        0.0       )   -- interior, period-1, all maxIter
canonical   : c = ( 0.0016437, -0.8224676 )   -- (prior reports) minibrot interior
\end{sourcecode}

\paragraph{Instrumentation neutrality.} All numbers below are \texttt{viz}=0
timing runs. The \texttt{viz}=3 animations (next section) were generated
separately, because \texttt{viz}=3 is \emph{not} performance-neutral for
trivial-region workloads: misiurewicz compute was 2.37~s at \texttt{viz}=0 but
\textbf{66.5~s at \texttt{viz}=3}, since the per-region \texttt{VizRect}
registration is a fixed cost that dominates when regions are $\sim$0.01~ms (it is
negligible for the expensive-region points: outside $16.95\to16.89$~s, inside
$\approx$76~s either way). The ``performance-neutral'' claim of
\texttt{08-region-metrics-viz.tex} was validated only on expensive-region frames.

\section{Analysis}

\subsection{Cost is content-driven and spans 32$\times$ at fixed parameters}

The same binary and the same \texttt{diffT}=0.1 produce walls from 2.37~s
(misiurewicz) to 76.28~s (inside), a 32$\times$ range (Table~\ref{tab:char}).
The reason is that total compute is $\sum_{\text{frames}}\sum_{\text{pixels}}$
(iterations to escape or \texttt{maxIter}), and the zoom target dictates how much
in-set or slow-escape area the path sweeps. The misiurewicz spiral is mostly
fast-escape exterior (red in Fig.~\ref{fig:regimes}) with only thin filaments, so
its 3{,}312 GPU leaves average 0.010~ms; the inside target is solid interior, so
every pixel runs to \texttt{maxIter} and its 573 GPU leaves average 127~ms ---
four orders of magnitude more per region. The implication for the study is that
``the workload'' is not a single number: any scheduling claim must state which
regime it was measured in.

\subsection{The dynamic schedule is robust in every regime}

The per-thread CPU spread is 0.2--1.4\,\% across all four points
(Table~\ref{tab:char}) --- tighter, even, than the canonical run's 3.8\,\%
(\texttt{04-postfix-profile.tex}). The mechanism is the one established in
reports~09 and~12: with thousands of fine-grained leaves over 12~threads
($\sim$330--530 leaves per thread here), the tail is always well-filled, so the
shared min-max queue keeps every worker busy until the work is gone. This is the
central load-balancing result: \emph{dynamic balance is already a solved problem
on this workload, in every regime tested}. The corollary --- important for the
planned static-load-balancing experiment --- is that static decomposition can at
best \emph{match} the dynamic queue here (the headroom is the $<$1.4\,\% spread),
and could only win in a latency-bound configuration the queue is not stressed by.

\subsection{The binding resource shifts from the CPU pool to the GPU}

The wall is set by a different resource in each regime: GPU utilisation runs
1.4\,\% (misiurewicz), 79\,\% (outside), 95\,\% (seahorse), 96\,\% (inside)
(Table~\ref{tab:route}). The light regimes are trivially CPU-pool-bound --- there
is so little GPU-worthy work that thread~0 is idle 98\,\% of the time. The heavy
regimes are GPU-saturated: affinity routes the few large coherent tiles to the
GPU (seahorse 28\,\% of leaves, inside just 15\,\%, but 58.7~ms and 127~ms each),
and the GPU fills to 95--96\,\%. This is the same $\sim$91\,\% saturation ceiling
report~12 hit on the canonical point, now confirmed to \emph{generalise} across
boundary and interior targets. The implication is that on heavy frames the GPU,
not the schedule, is the wall --- and the next lever is work-reduction, not
scheduling.

\subsection{The seahorse outliers: a saturated GPU overflows main-cardioid interior to the CPU}

Seahorse is the only regime with outliers --- eight of them, all
\texttt{cardioidBulb}=1 (Table~\ref{tab:char}, §3.3). They are the largest class
of region ($960\times540$, depth~1), which affinity \emph{tries} to send to the
GPU; but the GPU is already 95\,\% busy with other large tiles, so these eight
overflow to CPU threads at 17--18.7~s each. Per-thread balance still holds at
0.9\,\% because the run is throughput-bound: $8\times18$~s~$\approx$~144~s of
outlier work overlaps hundreds of other regions across the 69.8~s wall, so no
single region binds. The decisive contrast with the canonical point is the
membership flag: these are main-cardioid interiors, so unlike the canonical
minibrot (\texttt{cardioidBulb}=0, on which the cheap test cannot fire) the exact
cardioid/bulb certificate of \texttt{08-region-metrics-viz.tex} \emph{would}
constant-fill them in O(1). Seahorse is therefore the first concrete,
measured beneficiary of that certificate.

\subsection{Uniform interior produces zero outliers}

Inside is the heaviest run (76.3~s, every pixel to \texttt{maxIter}) yet has
\emph{no} outliers (Table~\ref{tab:char}). The mechanism is two-fold: a uniform
interior has zero corner spread everywhere, so the heuristic subdivides only down
to the \texttt{pixelSizeThresh} floor, producing many equal medium tiles rather
than one giant one; and GPU affinity then takes the largest of those (127~ms
avg), so the biggest piece any CPU thread ever sees stays under 5~s. Maximal
total work does not imply a tail --- the tail is a \emph{boundary} phenomenon
(seahorse, canonical), not a \emph{volume} one.

\section{Recommendations}

\begin{enumerate}
    \item \textbf{Use this as the baseline for the static-load-balancing study.}
    Dynamic balance is $<$1.4\,\% spread in all four regimes, so static
    decomposition (block vs.\ cyclic vs.\ block-cyclic) should be measured for
    \emph{parity at lower overhead}, not for a wall win --- and specifically in a
    latency-bound corner (CPU-only, or GPU-starved) where a single region can
    bind the critical path, which never happens here.

    \item \textbf{Implement the cardioid/period-2 bulb certificate; seahorse is
    its test case.} The eight \texttt{cardioidBulb}=1 outliers ($\approx$144~s of
    CPU work) would be constant-filled in O(1). Expected wall effect is bounded by
    GPU saturation (the relieved CPU work must still beat the 66.2~s GPU floor),
    so target the CPU-pool-bound and latency-bound configurations where it pays
    most. It does \emph{not} help the canonical minibrot (\texttt{cardioidBulb}=0).

    \item \textbf{Treat the 95--96\,\% GPU saturation as a general ceiling.} It
    reproduced on both heavy targets, confirming report~12: past it, only
    work-reduction (periodicity checking in \texttt{diverge()}) shrinks the wall.

    \item \textbf{Fix the filename-prefix buffer} (\texttt{main.cpp:201},
    \texttt{char imageFilePrefix[MAXFNAME-8]}, $\approx$42~B, read by
    \texttt{fin >> imageFilePrefix}): prefixes $\ge$42 chars truncate silently
    (the \texttt{viz\_misiurewicz/} overflow seen here). A \texttt{char[256]} or
    \texttt{snprintf} bound fixes it; it does not affect any measurement.
\end{enumerate}

\section{Conclusion}

Profiling the GPU-affinity binary across four zoom regimes shows that end-to-end
cost varies 32$\times$ (2.37--76.3~s) while the dynamic schedule stays balanced to
under 1.4\,\% everywhere: dynamic load balancing is already solved on this
workload, and the binding resource is content, not scheduling. In light regimes
the CPU pool binds trivially; in heavy regimes the GPU saturates at 95--96\,\%,
the same ceiling report~12 found, now shown to generalise. The seahorse boundary
target reproduces the interior outlier eightfold and identifies it as
\texttt{cardioidBulb}=1 main-cardioid interior overflowing a saturated GPU ---
the first measured case the cheap cardioid certificate would relieve, in contrast
to the canonical \texttt{cardioidBulb}=0 minibrot. The outliers never bind the
throughput-bound wall. The result reframes the remaining work: scheduling is
exhausted, and the levers are an interior certificate (for boundary tails) and
work-reduction (for the GPU-saturated volume).
